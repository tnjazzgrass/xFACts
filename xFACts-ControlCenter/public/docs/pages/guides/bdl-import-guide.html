BDL Import — User Guide

A step-by-step walkthrough for importing bulk data into Debt Manager
Tools • BDL Import Guide
The BDL Import wizard lets you upload a vendor data file, map its columns to Debt Manager fields, validate the data, and submit it as a Bulk Data Load — all from your browser. No more copy-pasting into Access tables. No more hardcoded XML. Just point, click, and let the system do the heavy lifting.

This guide walks through every screen of the 5-step wizard. Whether you’re importing phone numbers, tagging consumers, or updating account data, the workflow is the same. If you’ve used the old Access toolkit, think of this as that — minus the 90 MB file size and the existential dread of opening a shared database.

Access: The BDL Import page is available to the Applications & Integration team (full access to all entity types) and the Business Intelligence team (access to department-specific entity types configured by an administrator). Page access is controlled via RBAC; entity and field access are controlled per department.

3 Select Entity Types

The i icon on each card opens a field info modal showing all available fields for that entity type. This is a quick way to check what data the entity expects before selecting it. The modal shows display names in white, descriptions in grey, and import guidance (when populated) in amber italic.

Entity types with a FIXED VALUE badge use a simplified mapping interface — instead of mapping file columns to fields, you’ll enter fixed values that apply to every row. This is common for tag entities where every record gets the same tag name.

Multi-entity workflow: When you select multiple entities, Step 4 walks through each one individually — map then validate for Entity 1, then map then validate for Entity 2, and so on. Progress dots at the top of Step 4 show where you are in the sequence. All entity states are preserved if you navigate back.

4 Map & Validate

This is where the real work happens. For each selected entity type, you’ll map columns from your file to BDL fields, then validate the data. The wizard handles one entity at a time — when you complete mapping and validation for the first entity, it automatically transitions to the next.

When you’ve selected multiple entities, progress dots at the top show where you are:

Entity 2 of 3 — green dots are complete, blue is current, grey is upcoming.

File-Mapped Entities

For most entity types, you’ll see a two-panel layout: Source Columns (your file) on the left, and Target Fields (BDL fields) on the right.

First things first: select your identifier column. Before any mapping is enabled, you must tell the system which column in your file contains the consumer or account agency ID. This is the column DM uses to locate the right record. A dropdown at the top of the mapping area lets you pick it.

Once the identifier is set, drag source column chips from the left panel and drop them onto target field chips in the right panel. Or simply click a source chip, then click the target chip you want to map it to. Mapped pairs display together as combined chips showing the mapping relationship.

Visual indicators on target field chips:

* Asterisk (red): This field is required for a successful import. If it’s not mapped and has no value, the import will either fail or silently do nothing. Don’t ignore these.

∅ Null symbol (amber): This field supports nullification. Clicking it toggles “blanket nullify” mode, which explicitly clears this field in DM for every row. Useful when you want to wipe a field value across a batch. Empty values in mapped columns also trigger per-row nullification automatically.

Fields may also show an amber import guidance tip when hovered or clicked — these are operational notes from the team about how the field behaves (e.g., “Required numeric value — enter 0 if not in source data”).

Validation

After mapping, click Next (or the system auto-advances) to validate the data. The system stages your data on the server, fetches reference values from DM, and checks every row against the field rules. Issues are presented as accordion cards — one category at a time, most critical first.

Issue types and what you can do:

Required Empty (red) — A required field has no value in one or more rows. You can Fill with a value (applies the same value to all empty rows for that field) or Skip the affected rows entirely. The amber tip below the description is the import guidance — operational advice about what value to use.

Invalid Lookup (red) — Values in the file don’t match DM’s reference tables. You can Replace the invalid values with a valid one (shows a typeahead picker) or Skip those rows.

Max Length and Data Type (blue/informational) — These are warnings about values that exceed field length limits or don’t match the expected data type. Informational only — they won’t block the import but may cause DM to truncate or reject individual records.

Cascading validation: After each fill or skip action, the system automatically re-validates. Skipping rows for one field removes those rows from subsequent checks, so the counts may change as you work through the issues. The UI disables interaction briefly during re-validation to prevent race conditions.

Row alignment for multi-entity imports: When importing multiple entity types from the same file, skipping rows in one entity automatically aligns the other entities — rows skipped for PHONE are also skipped for CONSUMER_TAG, keeping the record sets consistent.

5 Review & Execute

The final step shows a summary of everything you’re about to import, with one tab per entity type. Each tab shows the entity name, environment, row count (after skips), and mapped fields.

XML Preview: Click the preview button to see the exact XML that will be sent to DM. This is the same structure the system generates — headers, wrapper elements, and all your data rows as XML entries. Useful for verifying the output before pulling the trigger, especially on your first import of a new entity type.

Jira Ticket (optional): If this import is associated with a Jira ticket, enter the ticket number (e.g., SD-1234) before executing. The system will automatically generate a consolidated AR log linking all imported records to the ticket. This creates an audit trail in DM that shows up on consumer accounts.

Executing: Click Execute Import to submit. Each entity type is submitted independently — the system builds the XML, writes the file to the DM file share, registers it via the DM REST API, and triggers the import. If one entity fails, the others are unaffected.

Promote to Production

If you ran a successful import on TEST or STAGE, a Promote to Production button appears in the results. This re-submits the same import to PROD without requiring you to walk through the wizard again — same file, same mapping, same data. The system rebuilds the XML targeting the production environment and submits it fresh.

▲ Promote to Production
Available after a successful TEST or STAGE import.

A cooldown timer prevents accidental double-submissions. After promoting, the button is disabled for a configured period. The environment badge updates to PROD to reflect the new target.

Promote is optional. You can always go back to Step 1, select PROD, and run the wizard from scratch. Promote just saves you the round trip when the data has already been validated.

Mapping Templates

If you frequently import files with the same column layout, templates save you from re-mapping every time. Templates store the column-to-field mapping for a specific entity type and can be applied to new imports with a single click.

The template panel lives in the right column below the step guide. When you’re on the mapping screen (Step 4), it shows all saved templates for the current entity type.

Loading a template: Click a template name to preview the stored mappings. If the template’s source columns match your file’s column headers, click Apply and the mapping is done instantly. If some columns don’t match (e.g., the file has different column names), the template applies what it can and you map the rest manually.

Saving a template: After you’ve completed a mapping, click Save Current Mapping as Template at the bottom of the template panel. Give it a descriptive name (e.g., “Acme Phone Export”) and an optional description. The template is tied to the entity type and available to all users.

Managing templates: Template creators and administrators can edit or deactivate templates. Deactivated templates are hidden from the list but not deleted — they can be reactivated if needed.

Tips & Troubleshooting

The Right Column

The right side of the page has two sections that stay visible throughout the wizard. The step guide panel at the top updates automatically as you move through steps, showing context-specific tips and instructions. The template panel below it is relevant during Step 4 (mapping).

Going Back

The Back button works throughout the wizard and preserves your state. If you’re on entity 2 of 3 in the mapping step, Back takes you to entity 1 (not all the way back to Step 3). From entity 1, Back takes you to Step 3 (entity selection). Your mappings and validation results are preserved on back navigation — nothing is lost.

Clicking the Stepper

You can click completed steps in the stepper bar to jump back directly. You cannot click forward to steps you haven’t reached yet. Completed steps show a green checkmark.

Staging Table Cleanup

When you validate data, the system creates a temporary staging table on the server. These tables are automatically cleaned up after 48 hours. You don’t need to do anything — but if you’re curious, the staging table name follows the pattern Staging.BDL_{entity}_{user}_{timestamp}.

What Happens After You Execute

The import file lands on the DM file share, gets registered via the REST API, and the import is triggered. From there, DM’s internal processing takes over. You can monitor import progress on the Batch Monitoring page under the BDL section, where the BDL collector tracks files through staging and completion.

Error Recovery

If an import fails (API error, file write failure, etc.), the error is displayed in the results pane. DM requires a new file with a new name for retry — you cannot re-submit the same filename. Go back to Step 1 and start fresh; the system generates a unique batch ID for each submission.

File Size Limits

The practical limit is approximately 250,000 rows per import. This is a DM processing constraint, not a system limitation. For larger datasets, split the file into batches and run them sequentially.

Access Control

Entity types and individual fields are controlled per department. If you don’t see an entity type you expect, your department may not have been granted access. Contact the Applications team to request access to additional entity types or fields. Admin-tier users (Apps/Int team) bypass all access restrictions and see everything.
