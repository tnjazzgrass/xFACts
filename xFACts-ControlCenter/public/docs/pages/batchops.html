Batch Monitoring

Watching the pipeline so you don't have to stare at it
Batch Monitoring watches the Debt Manager batch pipeline — New Business file uploads, Payment posting, and the nightly maintenance window. It doesn't process batches. It doesn't fix batches. It watches them, and when something goes sideways, it tells you about it before anyone else notices. Think of it as the person in the control room who actually pays attention to the monitors.

The Problem

Clients send us data. That data arrives as files. Those files become batches. Those batches go through a series of processing stages — uploading, releasing, merging, posting — until eventually the accounts land in the system and everyone's happy.

Eventually.

Because somewhere in that chain, things can go wrong. An upload fails. A batch gets stuck in RELEASED and never starts merging. A payment file sits in INPROCESS for three hours while everyone assumes someone else is watching it. And nobody finds out until the next morning when the reports look wrong and a client asks why their data isn't in the system.

That's a domino effect. Accounts that didn't load. Reports that show the wrong numbers. Someone in Operations having a very long day because nobody noticed the failure until it was too late.

BatchOps catches failures early — before they become situations.

Collect → Track → Evaluate → Alert — every 5 minutes

BatchOps runs two independent collectors, each watching a different batch type. Every 5 minutes, each collector reads the Debt Manager source system through the AG secondary replica, updates its tracking table with the latest state, evaluates alert conditions, and fires notifications when something needs attention.

The New Business collector (Collect-NBBatchStatus.ps1) watches file uploads through the merge pipeline. It tracks 8 distinct alert conditions — upload failures, release failures, stalled merges, queue waits, and more. Each condition has its own deduplication strategy and configurable routing to Teams, Jira, or both.

The Payment collector (Collect-PMTBatchStatus.ps1) watches payment batch posting across all batch types — IMPORT, MANUAL, REVERSAL, and REAPPLY. It tracks journal-level progress during processing and alerts on terminal failures. PARTIAL batches get special handling because they can be re-fired in DM and shouldn't be treated as permanently failed.

The pre-maintenance summary (Send-OpenBatchSummary.ps1) runs once daily before the maintenance window. It queries DM directly for anything still in flight and sends a color-coded Adaptive Card to Teams. Green means safe to proceed. Yellow means batches are still processing — maybe hold off on that reboot.

It's the automated equivalent of yelling "IS ANYONE STILL IN THE BUILDING?" before locking the doors.

Two Pipelines, One Pattern

New Business and Payments are fundamentally different batch types in Debt Manager, but BatchOps monitors them with the same architectural pattern: collect from DM, track in xFACts, evaluate conditions, alert when needed.

New Business

File uploads that go through upload → release → merge. The interesting part is the dual state machine — a batch has both a batch status (tracking the upload/release lifecycle) and a merge status (tracking what happens after release). Most of the action is in the merge status, because nearly all batches reach RELEASED just fine. It's the merge that gets complicated.

Stall detection uses merge log activity — if the log stops updating but the merge isn't done, something's stuck. Eight alert conditions cover everything from upload failures to queue waits to that awkward state where a batch was released but nothing happened for six hours.

Payments

Payment batches that go through import → processing → posting. The interesting part here is journal-level progress tracking. While a batch is INPROCESS, xFACts counts how many payment journals have posted vs. how many are still pending. This gives real-time visibility into posting progress that DM's batch header status alone doesn't provide.

Alert conditions focus on terminal failures — IMPORTFAILED, FAILED, REVERSALFAILED. PARTIAL gets a WARNING because it's non-terminal; the batch can be re-fired in DM, and the collector watches for that recovery automatically.

Both collectors run independently under the orchestrator on 5-minute cycles. They don't know about each other and they don't need to. Each one owns its tracking table, its alert evaluation, and its own set of GlobalConfig settings.

The Alert System

Not all problems are created equal. An upload failure needs a Jira ticket right now. A batch sitting in a queue for 5 hours needs a Teams heads-up. A batch that's been in RELEASED for 8 hours without any merge activity needs someone to notice.

Each alert condition has three configurable dimensions:

DimensionWhat It Controls
Routing Where the alert goes: Teams only (1), Jira only (2), both (3), or disabled (0). Uses bitwise flags so each condition is independently configurable.
Thresholds How long to wait before alerting. A merge stall after 1 hour of inactivity (12 polling cycles). A queue wait after 5 hours. Each threshold is tuned to the real-world behavior of that condition.
Deduplication How to avoid flooding. Three strategies: one alert per episode (batch_id only), one alert per stall point (batch_id + log position), or daily re-alerting for ongoing conditions (batch_id + date).

A master switch (nb_alerting_enabled / pmt_alerting_enabled) controls whether any alerts fire at all. Individual routing settings are only evaluated if the master switch is on. This makes it easy to silence everything during planned maintenance without touching individual settings.

Why bitwise routing? A single integer per condition controls two independent outputs. 3 means both Teams and Jira. 1 means Teams only. The script checks routing -band 1 for Teams and routing -band 2 for Jira. Clean, compact, and no need for separate "enable_teams" and "enable_jira" settings per condition.

The Pre-Maintenance Summary

Every evening before the maintenance window, a color-coded Adaptive Card shows up in Teams. It's the batch pipeline's status report — a quick visual answer to "is anything still running?"

Section ColorMeaning
Green All clear. No batches in flight for this type. Safe to proceed.
Yellow Batches still processing. Details listed with batch IDs, types, statuses, and timestamps.
Gray Not monitored yet. BDL and Notice Processing sections appear as placeholders for future expansion.

The card is built modularly — each batch type has its own check function that returns a standardized result. Adding a new batch type means writing one function, calling it in the main flow, and the card builder picks it up automatically. The architecture is ready for expansion even though only NB and PMT are active today.

Unlike the collectors, the summary script queries DM directly rather than reading the tracking tables. This means it reflects the live state at query time, not the last polling snapshot. A subtle distinction that matters when you're making a "safe to reboot?" decision.

Source System Decoupling

BatchOps monitors Debt Manager, but it doesn't depend on Debt Manager's internal logic. The collectors read DM's tables and status codes, but all tracking, stall detection, and alerting logic lives in xFACts. If DM changes how it processes batches, BatchOps keeps working as long as the status codes still mean what they mean.

This is deliberate. DM is the system of record for batch data. xFACts is the system of record for monitoring that data. The two systems share a read-only boundary — xFACts reads from DM's secondary replica and never writes back. If the replica is offline, the collectors log the failure and try again next cycle. No data corruption. No interference with production.

AG replica awareness: Both collectors read from the Availability Group secondary replica by default. The AG configuration (which replica to target) comes from GlobalConfig, and a ForceSourceServer parameter can override it for testing. If the secondary is unavailable, the script fails gracefully and retries on the next orchestrator cycle.

The Control Center View

The BatchOps page in the Control Center shows the current state of the batch pipeline at a glance. Active batches with their current status, stall indicators, and completion tracking. The data comes directly from the tracking tables that the collectors maintain.

Collector health is visible through the BatchOps.Status dashboard — each collector and the summary script have their own row showing last run time, processing status, and whether the last execution succeeded or failed. If a collector shows IDLE with a last_run_dttm that's more than 10 minutes old, something's wrong with the orchestrator schedule.

Historical analysis is available through the tracking tables. Monthly batch volumes, average completion times, failure rates by batch type — all queryable from the data the collectors maintain as part of their normal operation. No extra collection step needed.

Troubleshooting

"A batch failed but I didn't get an alert."

Start with the master switch: is nb_alerting_enabled (or pmt_alerting_enabled) set to 1 in GlobalConfig? Then check the individual condition's routing value — 0 means disabled. Then check the deduplication trail: has an alert for this batch already been delivered? Check Teams.RequestLog and Jira.RequestLog for the trigger_type and trigger_value combination.

"I'm getting duplicate alerts for the same batch."

Check the orchestrator schedule. If the collector runs more frequently than expected (overlapping cycles), deduplication checks may not catch the duplicate in time. The orchestrator's fire-and-forget model prevents overlapping execution, but if someone is running the script manually alongside the scheduled cycle, that's two instances evaluating the same conditions.

"The pre-maintenance card didn't send."

Check BatchOps.Status for the Send-OpenBatchSummary row — did the script run? Check Teams.AlertQueue for a row with trigger_type = 'OpenBatchSummary'. If it's there but stuck in Pending, the Teams processor may not be running. If it's not there, the script didn't get far enough to insert it.

"A PARTIAL payment batch shows as complete."

If you're on a version before the PARTIAL recovery fix, the batch may have been marked complete and then re-fired in DM. The collector now detects this and resets the completion fields automatically. For historical data, manually clear is_complete, completed_status, completed_dttm, and alert_count on the affected row.

"Stall detection is firing false positives."

The stall thresholds are tuned to typical processing times, but large batches or slow DM periods can legitimately exceed them. Increase the relevant threshold in GlobalConfig — nb_stall_poll_threshold for merge stalls, nb_upload_stall_minutes for upload stalls, etc. The defaults are conservative intentionally.

The Bottom Line

Batch processing is one of those things where small problems become big problems if nobody's watching. A failed upload at 2 AM turns into a missing report at 8 AM turns into an unhappy client by noon.

BatchOps watches so you don't have to. Upload failures get Jira tickets. Stalled merges get Teams alerts. Terminal payment failures get flagged immediately. And every evening, a simple card tells you whether it's safe to start maintenance or whether you should wait five more minutes.

Two collectors. Three tables. One pre-maintenance card. And the confidence that if something goes wrong in the batch pipeline, you'll know about it before anyone asks.
