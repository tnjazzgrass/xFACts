About This Page

The Server Health page is the most information-dense page in the Control Center.

It provides real-time and snapshot metrics from the Activity and Disk components of the ServerOps schema, and it can be overwhelming at first glance.

This guide outlines each section, indicator, and clickable element so you know exactly what you’re looking at.

1Server Selection Tabs

The row of mini-cards in the center of the header represents each monitored SQL Server. Each card shows the server name and a live CPU utilization gauge. Click a server to switch the entire page to that server’s metrics.

The currently selected server has a highlighted border. All metrics, charts, and panels update to reflect the selected server. You can switch servers at any time — the page rebuilds instantly.

The CPU gauge on each server tab gives you a quick health-check across the entire fleet without clicking into each one. If one server’s gauge is noticeably higher than the others, that’s worth investigating.

2Engine Indicators

The small cards in the upper right represent the orchestrator processes that feed this page with data. Server Health has four: DMV (performance metrics), XE (Extended Events), Disk (disk space collection), and Summary (disk health summary alerts).

Each indicator shows a colored bar and a countdown to the next run. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in — which could mean the engine itself needs attention.

These update independently of the page’s refresh cycle via a persistent WebSocket connection. They’re your real-time pulse on the data pipeline behind the page.

3Memory

Four metric cards that tell you how well SQL Server is managing its memory.

Page Life Expectancy (PLE) — How many seconds a data page stays in memory before being pushed out. Higher is better. When this drops, SQL Server is under memory pressure and constantly swapping data in and out. Click for historical trend (24h / 7d / 30d).

Buffer Cache Hit Ratio — The percentage of data requests served from memory rather than disk. Should be 99% or higher. A drop means queries are hitting disk far more than they should. Click for trend.

Memory Grants Pending — Queries waiting in line for memory. Should be zero. Non-zero means queries are queuing up because there isn’t enough memory to go around. Click for trend.

Lazy Writes/sec — How fast SQL Server is flushing pages to disk because it needs the memory. Lower is better. Occasional lazy writes are normal; sustained high values indicate pressure.

Each card changes color based on configurable thresholds — green when healthy, yellow for warning, orange for critical, red for crisis.

4Connections

Four metric cards tracking who and what is connected to the server.

Active Connections — Every active session connected to the instance. Click for trend chart showing connection patterns over time.

JDBC Connections — Connections specifically from Debt Manager (identified by the JDBC driver). This is the primary application workload. Click for trend.

Zombie Connections — JDBC sessions sitting idle beyond threshold with no open transaction. Using connection slots but doing nothing useful. Clicking opens a confirmation dialog to kill them all at once. Yes, the button says “Double Tap Them.” Yes, there are zombie emojis.

Zombie killing terminates actual database sessions. The threshold is conservative enough that false positives are rare, but only use this when you’re confident the idle connections are genuinely abandoned. Click marker 10 on the mockup to see the confirmation dialog.

Open Transactions — Sessions with uncommitted transactions idle beyond threshold. Potentially dangerous because they hold locks that can block other processes. Click to open a slide panel with session details and a button to copy a KILL script.

5Current Activity

Four metric cards showing what’s happening on the server right now.

Blocked Sessions — Sessions currently waiting on another session to release a lock. When non-zero, someone is blocking someone else. Click to open a panel showing the blocking chain — who’s blocking whom, what query they’re running, and how long. Includes a copy-to-clipboard KILL script for the lead blocker.

Lead Blocker — If blocking is occurring, this identifies the SPID at the head of the chain — the session everyone else is waiting on. Shows “None” when there’s no blocking. When blocking is active, the card shows the SPID and how many sessions it’s holding up.

Longest Wait — The duration of the longest currently active wait. Displayed in seconds or minutes depending on the value. When this is zero, nobody is waiting for anything — the server is moving freely.

Active Requests — Sessions actively executing a query right now. The subtitle breaks these down by status: running (green), runnable (yellow), and suspended (red). Click for a detail panel showing what each session is running, how long, and resource consumption. Includes a manual refresh button since this is a point-in-time snapshot.

6Extended Events Activity

Counts of events captured by SQL Server’s Extended Events sessions within a configurable time window. The default is 15 minutes — change it with the gear icon next to the section header (5m, 15m, 30m, 60m).

This section refreshes when the XE collector completes (lightning bolt badge), not on the live page interval.

Long Running Queries — Queries exceeding the duration threshold. Click for query text, database, duration, and resource stats.

Blocking — Sessions that blocked another long enough to trigger the threshold. Click for blocker and victim query details.

Deadlocks — Mutual blocking that SQL Server resolved by killing a session. Click for deadlock details.

Linked Server Inbound / Outbound — Cross-server query traffic. Useful for understanding integration patterns.

AG Events — Availability Group state and role changes from the AlwaysOn health session.

System Health — SQL Server’s built-in diagnostics — memory pressure, non-yielding schedulers, severe errors.

Non-zero counts aren’t necessarily problems. Long running queries happen during legitimate operations. The value is in the trend — a sudden spike in blocking events when nothing special is scheduled is worth investigating.

7Server Info

A summary panel in the right column showing the selected server’s configuration: SQL Server version, edition, total memory, CPU count, uptime, and AG replica role (Primary or Secondary). Refreshes when you select a different server (swap arrows badge).

8Disk Space

Visual bars showing used space on each drive of the selected server. Color-coded against per-drive thresholds — green when healthy, yellow when approaching, red when below.

Refreshes when the disk collector process completes (lightning bolt badge), not on the live interval.

9DMPRODAG Health

The Availability Group panel shows the current primary and secondary replicas and overall synchronization health. Click a replica name to switch the dashboard to that server and open a detail slideout with queue sizes, redo rates, and last hardened LSN timestamps.

Refreshes on the live page interval (pulsing dot badge).

10Zombie Eradication

When the Zombie Connections metric card in the Connections section shows a non-zero count, clicking it opens a confirmation dialog offering to kill all zombie sessions at once.

The dialog shows the zombie count, the idle threshold (configurable in GlobalConfig), and the target server. Two buttons: “Never Mind” to back out, and “Double Tap Them” to execute the kill. Yes, there are zombie emojis. No, we will not apologize for them.

Behind the scenes, the kill operation runs KILL commands against each identified zombie SPID. Results are logged. The Connections section refreshes automatically after execution to confirm the zombies are gone.

This terminates actual database sessions. The idle threshold is conservative enough that false positives are rare, but verify the zombie count looks reasonable before clicking. If a session appears idle but is actually waiting on a long-running distributed transaction, killing it could cause a rollback.
