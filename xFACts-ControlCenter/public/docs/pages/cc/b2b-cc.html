About This Page

The IBM/B2B Pipeline page is the live window into IBM Sterling B2B Integrator — the middleware that moves files between the outside world and Debt Manager.

The left side is today and now: a pulse of the day’s counts up top, and a live view of the runs currently in motion below. The right side is the full run history, organized year → month → day, with search and filters that open a runs list you can drill into.

This guide walks through every section, the two status vocabularies you’ll see, and the slide-out panels that open when you click into a run.

1Engine Indicator

One engine card in the upper right represents the process behind this page: the B2B collector (Collect-B2BPipeline.ps1). It mirrors Sterling’s pipeline runs into the tracking tables, classifies each one, and enriches failed runs with their captured fault report.

A green bar means the last run succeeded; red means it failed. The countdown shows time until the next scheduled run. This card updates on its own, independently of the page’s refresh cycle.

The same shared engine-indicator appears on every monitoring page in the Control Center, so this card behaves identically to the ones on Batch Monitoring, Server Health, and the rest.

2Pulse — Live Row

Today’s Pipeline Pulse is a row of six tiles giving the day at a glance. The top three describe what’s happening right now and are sourced from the same live feed as the activity panel below, so they can’t lag behind it.

Runs Today — the total number of pipeline runs that have started today.

In Flight — runs currently executing in Sterling.

Awaiting DM — runs that finished on the Sterling side and are waiting for Debt Manager to confirm the handoff landed.

These three are informational — they are not clickable. The In Flight and Awaiting DM counts always match the rows you see in the Live Pipeline Activity panel directly below.

3Pulse — Clickable Row

The bottom three tiles describe how today’s finished runs turned out. Unlike the top row, these are drawn from the tracked history table — and each one is clickable.

Success (green), No Action (neutral), and Failed (red) each show today’s count for that outcome. Clicking any of them opens the runs list (a slide-out panel) pre-filtered to exactly that outcome for today — so the runs you land on match the number on the tile.

Clicking Failed is the fastest way to see today’s problems: it drops you straight into the list of failed runs, each of which you can open for its full detail and fault report.

“No Action” is a normal, healthy outcome — it counts runs that correctly checked for work and found nothing to do. A large No Action count is expected, not a problem.

4Live Pipeline Activity

A live table of the runs currently in motion, read straight from the source so it’s genuinely real-time. Each row shows the process-type badge, the Run ID, the sequence id, the client, a status badge, and the run’s age.

The status here uses the detailed classification — you’ll see specific states like “In Flight” and “Awaiting DM” rather than the coarser Sterling status used elsewhere. A run with no process type is a parent/dispatcher run and badges as DISPATCHER.

Rows here are not clickable. Because this panel runs ahead of the collector, a run shown here may not have been recorded into the history table yet — so its full detail would be empty. Once it’s collected, it appears in the Run History on the right, where its row is clickable.

5Run History

The full history of every pipeline run, organized as a collapsible year → month → day tree. It refreshes when the collector completes a cycle (the lightning-bolt badge), since that’s the only time new history appears.

Every level is bucketed by the five coarse Sterling statuses — Runs, Success, Failed, No Action, In Progress — plus an Undefined column and an average duration. A dash means zero, which keeps the grid quiet and lets the real numbers stand out.

The counts here use the Sterling status — the “how did this end in Sterling” view. That’s the coarser of the two status vocabularies on the page; the detailed classification appears when you open an individual run.

6Filter Bar & Runs List

The filter bar above the tree lets you find specific runs. It doesn’t filter the tree itself — instead, Search opens a runs list (a slide-out panel) matching whatever you entered.

You can search by Run ID (exact match), by client (partial match), and narrow by status, process type, and a date range. The Type dropdown only lists types that actually appear in the data, so it never offers an empty result. Reset clears everything.

The runs list is paged, and each run in it is clickable — opening its full detail. This same list is what the clickable pulse tiles open, just pre-filled with their filter.

7Year / Month Tree

Click a year to expand its months; click a month to expand its days. The tree opens one branch at a time — opening a new year closes the previous one, and opening a new month closes the previous one — so it never grows into an unreadable wall.

Year rows carry the roll-up totals for the whole year; month rows break it down by month; both use the same status columns as the day rows below them.

8Day Rows & Run Detail

Each day row shows that date’s totals. Click one to open the day runs slide-out — a paged list of every run on that date, with its type, Run ID, start time, client, dispatcher, status, duration, and a Jira indicator where tickets exist.

Clicking any run — whether from a day list, the filtered runs list, or a pulse tile — opens the run-detail slide-out. That panel tells the run’s whole story: a header carrying both status readings (the Sterling result and the detailed classification), at-a-glance facts, and cards for its identity, timing, files, and any Jira tickets. The panel appears right away; the Files and Tickets cards load separately, so they may fill in a moment after the rest of the panel is already on screen.

If a run failed and Sterling produced an error report, the run-detail panel shows a fault summary with a View Full Report button. That opens the captured Sterling status report in a readable form — the specific translation or service error, right there, with no need to log into Sterling.

If you open a run that the collector hasn’t recorded yet (one still showing in Live Pipeline Activity), the panel doesn’t error out — it shows an “awaiting first collection” note alongside whatever live snapshot is available, then fills in completely once the run is collected.

9Schedules Modal

The Schedules button at the right of the Run History header opens a pop-up modal listing the Sterling schedules — the recurring jobs that drive the pipeline. It answers two everyday questions: what’s scheduled to run, and when. Each row shows the service (the workflow), a plain-English schedule (“Daily at 04:00,” “Mon-Fri at 14:00,” “Every 10 min...”), the timing pattern, how many times it runs per day, and whether it’s active or inactive.

The filter bar across the top is built for quickly narrowing the list. It groups into three: search plus pattern and status dropdowns; two checkboxes (runs today, and hide the system/utility schedules); and a “fires between” time window with a reset. Everything filters instantly as you change it — the whole schedule set is loaded once when the modal opens, so there’s no waiting between filters.

The fires between window is the useful one for planning maintenance. Set a start and end time and the list narrows to only the schedules that would fire in that window today — so an empty result means the window is clear. That’s the quick way to find a safe gap to restart a server without interrupting a scheduled job.

This is a read-only view today — it surfaces what Sterling has scheduled. The active/inactive state shown here is Sterling’s own; changing it isn’t done from this modal.
