About This Page

The Job/Flow Monitoring page tracks everything running through Debt Manager’s processing engine — scheduled overnight flows, ad-hoc daytime jobs, and the monitoring pipeline behind them.

It combines real-time queue visibility with historical execution data, stall detection, and administrative tools for managing flow configuration and task distribution.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicator

The single engine card in the upper right represents Monitor-JobFlow.ps1 — the orchestrator process that drives everything on this page. It runs on a fixed interval, querying Debt Manager’s processing tables and updating xFACts with flow status, job progress, stall detection, and missing flow checks.

The colored bar and countdown show the health and timing of the monitor. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in — which could mean the engine itself needs attention.

Most sections on this page refresh when this process completes (lightning bolt badges). The Live Activity section is the exception — it polls independently on its own timer because it queries Debt Manager directly for real-time queue state.

2Daily Summary

Three summary cards at the top give you the day’s vital signs at a glance.

Total Jobs — The count of all job executions today, across all flows and ad-hoc runs. This is the raw volume number.

Flows — How many distinct flows have run today, broken down by status: done (green), active (blue), and failed (red). If a flow ran more than once, the run count appears in parentheses. Click a flow in the list below for execution details.

Stall Events — The number of processing stall episodes detected today. A stall means the monitoring script observed no progress across the entire processing queue for multiple consecutive checks. Zero is the goal. Click this card to open a slideout with episode details — start/end times, duration, poll count, whether an alert was sent, and whether the stall crossed midnight.

Below the cards, the Flows section lists each flow that ran today with its flow code, friendly name, job count, success/failure breakdown, duration, and status badge. Click any flow row to open a slideout with the full job-level detail for that flow’s execution.

If ad-hoc jobs ran outside of any flow, an Ad Hoc Jobs link appears at the bottom. Click it to open a slideout with those executions.

3Live Activity

This section shows what’s happening in Debt Manager’s processing queue right now. It refreshes on the page’s live polling interval (pulsing dot badge) and queries Debt Manager directly — not from cached xFACts data.

Currently Executing — A table of every job actively running. Each row shows the job name, flow code (or “ad-hoc” if unaffiliated), a progress bar with record counts, success/failure tallies, estimated time remaining, processing rate, and log ID.

The Pending button in the section header shows how many jobs are queued and waiting to execute. The count badge updates live. Click it to open a slideout listing every pending job with its name, flow, and queued time.

During overnight processing, the Currently Executing table is your real-time window into what’s happening. If the table is empty and the Pending count is zero during expected processing hours, something may have stopped the queue — check the Stall Events card in Daily Summary.

4Process Status

A grid of eight small cards representing each step of the JobFlow monitoring pipeline. Each card shows the step name, an icon, the result count from the last run, and how long ago it executed. These update when the engine process completes (lightning bolt badge).

Config Sync — Synchronizes flow definitions between Debt Manager and xFACts. The count shows how many configuration differences (drifts) were detected. Zero means everything is in sync. A non-zero count turns the card yellow — click it to open the Flow Configuration modal for details.

Flow Detection — Identifies flows that have started executing since the last check. The count shows how many new flows were detected this cycle.

Completed Jobs Capture — Picks up jobs that finished since the last check and records their results. The count shows how many completions were captured.

Flow Progress — Updates the running totals for active flows — how many jobs completed, how many remain, current success/failure counts.

Flow State Transitions — Moves flows between states (detected, in progress, complete, validated, failed) based on their current job results and completion status.

Stall Detection — Checks whether any progress has been made since the last cycle. If the processing queue hasn’t moved, the stall counter increments. The card stays green at zero, turns yellow as the counter climbs, and turns red when it hits the configurable threshold. Hitting the threshold triggers an alert.

Flow Validation — Examines completed flows for hidden problems — jobs that didn’t run, jobs where every record failed, unexpected job counts. Flows that pass validation move to the VALIDATED state.

Missing Flows — Checks whether any scheduled flows haven’t started within their expected time window. A non-zero count means a flow that should have triggered hasn’t been seen yet.

The Process Status cards are a diagnostic view of the monitoring pipeline itself, not the processing it monitors. If a card shows a non-zero count, it doesn’t necessarily mean something is wrong with processing — it means the monitor found something to report. Check the Daily Summary and Live Activity sections for the actual processing state.

5Execution History

A collapsible tree showing the complete history of all flow executions. Organized by year, month, and day, with job counts and success/failure tallies at every level.

Click a year to expand its months. Each month row shows distinct flow count, total jobs, succeeded, and failed. Click a month to load its daily breakdown with day-of-week, date, flow count, and job results. Click a day to open a slideout with every flow and job that ran that day — full execution details, timings, and per-job results.

Status indicators on day rows give a visual summary: green means all jobs succeeded, yellow means some failures, red means all failed.

Use the history tree to spot patterns. If the same flow fails every Tuesday, or if job counts suddenly drop on a specific date, the drill-down will show you exactly what happened. The year-level totals also give you a quick sense of processing volume trends.

6App Server Tasks

A modal dialog accessed via the “App Server Tasks” button in the Process Status header. Shows a grid of all monitored flows and which application server each one is assigned to run on.

Each cell in the grid represents a flow/server combination. The currently assigned server is highlighted. Click a cell to stage a change — moving a flow from one server to another. Staged changes are tracked visually with color indicators showing the current assignment and the proposed new one.

After staging changes, an “Apply Changes” button appears. Clicking it opens a confirmation dialog listing every change before committing. Changes take effect on the next processing cycle.

Task reassignment is an administrative action that affects which application server runs each flow. Reassigning a flow during active processing could cause the flow to restart or duplicate work. Coordinate with the team before making changes during processing hours.

7Flow Configuration

A modal dialog accessed by clicking the Config Sync card in Process Status. Provides a view of each flow’s monitoring configuration — what schedule is expected, whether monitoring is active, and what alerts are enabled.

Select a flow from the dropdown to see its current configuration. Existing flows show their schedule type, expected start time, tolerance window, monitoring status, and alert settings. The display adapts based on flow type — scheduled flows show full timing details, variable flows show simplified settings, and on-demand flows show minimal configuration.

Flows already known to xFACts can be edited (schedule, monitoring, alerts) or deactivated. Deactivation sets an end date and disables monitoring without deleting the configuration.

New flows detected in Debt Manager but not yet configured in xFACts appear with a setup form. You choose the flow type (scheduled, variable, or on-demand), fill in the appropriate settings, and save. A confirmation dialog shows a summary of all changes before committing.

Flow Configuration is how xFACts knows what to expect. If a flow’s schedule changes in Debt Manager, updating it here ensures the Missing Flows check uses the right time window. A flow that isn’t configured won’t be monitored for missing runs.
