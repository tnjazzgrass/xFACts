About This Page

Platform Monitoring is the “are we the problem?” page. It measures the resource impact of xFACts itself on the SQL Server environment it monitors.

Unlike other dashboard pages, this one has no automatic refresh. Everything updates when you select a server or change the time range. It’s an analysis tool, not a live monitor.

The page answers two core questions: how much CPU is xFACts consuming, and how is the Control Center web interface performing?

1Narrative Summary

The colored strip below the header that tells the story of the data in a single sentence. It dynamically generates a plain-English summary combining CPU impact percentage, query count, heaviest process, and any alert conditions — all for the currently selected server and time range.

The left accent bar changes color with the CPU impact: green under 2%, yellow between 2–5%, and red over 5%. This gives you the headline before you read the details.

The narrative updates every time you select a different server or change the time range. It synthesizes data from multiple sources into one glanceable sentence — the idea is that you can read this strip alone and know whether xFACts is behaving itself.

2CPU Impact & Server Selection

The center column serves double duty: the hero gauge shows the headline number, and the server selectors below it control what the entire page displays.

Hero Gauge — A donut chart showing xFACts CPU consumption as a percentage of total server capacity. The calculation is straightforward: xFACts CPU time divided by total CPU capacity (cores × time period × 1,000ms). This is the same model Windows Task Manager uses. Below the percentage, a detail line shows raw CPU time consumed and core count.

Color thresholds: green under 2% (virtually invisible), yellow 2–5% (worth monitoring), red over 5% (investigate which processes are consuming resources).

ALL Pill — When active (teal highlight), the page shows aggregate data across all monitored servers. This is the default view.

Mini Gauges — Each small donut represents one server. The percentage and color show that server’s individual CPU impact. Click a mini gauge to filter the entire page — hero gauge, narrative, cards, tables, and chart all update to show only that server’s data. Click the same server again to return to the ALL view.

If the ALL view shows a healthy 0.12% but one mini gauge is yellow at 2.1%, click that server to drill into what’s different. The Process Breakdown table will show which processes are heavier on that specific server.

3Platform Performance

Six metric cards measuring the database activity generated by xFACts processes — the monitoring scripts, data collectors, and automation jobs that run on schedule. These reflect only xFACts activity, not total server activity.

Active Sessions — How many xFACts processes have active database connections right now. A real-time snapshot, not a time-range total. A handful is normal.

Total Queries — Total SQL queries xFACts executed during the selected time range. A high number alone isn’t concerning — what matters is how much CPU those queries consumed.

Avg Duration (ms) — Average time per query. Under 10ms means queries execute almost instantly.

Blocking Events — A dual-count card. “Blocked by others” means xFACts was the victim. “Caused by xFACts” means xFACts was holding locks that blocked other applications. Zero for both is the goal. Click to open a slideout with every blocking event: timestamp, server, direction, wait time, wait type, and the application names on both sides. Click any row in the slideout to expand the actual SQL queries from both the blocked and blocking sessions.

LRQ Crossovers — Queries that exceeded the configured duration threshold. Think of it as a speed trap. Click to open a slideout showing every long-running query: timestamp, server, process name, database, duration, CPU time, and logical reads. Click any row to see the full query text.

Open Transactions — Connections with uncommitted transactions (real-time snapshot). Zero is normal. Persistent nonzero values hold locks and can block other queries.

Every metric card has a ? icon. Click it for a detailed explanation of what that metric measures, how it’s calculated, and what values to watch for. The info modals are extensive — they’re the built-in documentation for the page.

4Control Center API

Six metric cards measuring the performance of the Control Center web interface itself — the dashboards and API endpoints you’re using right now. These are separate from database monitoring activity. The blue accent color distinguishes API metrics from platform performance metrics.

API Requests — Total HTTP requests handled during the time range. Every page load and auto-refresh generates several API calls.

API Req/Min — Average requests per minute. Higher numbers usually mean more people have dashboards open.

API Avg (ms) — Average response time. Under 50ms means pages load instantly.

API P95 (ms) — The 95th percentile response time — 95% of requests were faster than this value. More useful than the average because it reveals the worst typical experience.

API Users — Distinct users who accessed the Control Center during the time range. Click to open a slideout with a user-by-user breakdown showing request counts, distinct pages visited, and last active time. Summary statistics show total authenticated vs. unauthenticated requests (login page hits, expired sessions).

API Errors — Requests that returned HTTP 400 or higher. Zero is the goal. Click to open a slideout showing every error with endpoint, method, status code, user, and timing. Client timeouts (HTTP 408) from backgrounded browser tabs are shown in a separate collapsible section grouped by endpoint, since they’re typically harmless.

5Process Breakdown

A ranked table showing every xFACts process that ran queries during the selected time range. This is where you find out which process is responsible for the CPU number in the hero gauge.

Two percentage columns tell complementary stories. % xFACts shows each process’s share of total xFACts CPU — useful for comparing which processes are heaviest relative to each other. % Server puts it in real-world context — what percentage of the entire server’s capacity this individual process consumed. When every row shows values like 0.05%, 0.02%, 0.01%, that’s a clear indicator that no single process is contributing to performance issues.

Additional columns include query count, total CPU time, average query duration, and logical reads (data pages touched in memory). Column headers are sortable — click to reorder.

Hover over any row for a tooltip combining both percentage perspectives into a plain-English summary.

6CPU Impact Over Time

A dual-axis chart showing how xFACts CPU usage and query volume changed over the selected period.

The green line (left axis) tracks CPU percentage over time — the same metric as the hero gauge, but broken into time buckets. Hover over any point to see the exact values used in the calculation. The blue bars (right axis) show query count per bucket — these typically spike when scheduled collection processes run.

Together they tell a story: if query volume spikes but CPU stays flat, the queries are lightweight. If both spike together, a heavier process was running during that window.

Time range controls — Five buttons above the chart: 1h, 12h, 24h, 7d, and Custom. These control the analysis window for the entire page, not just the chart. Selecting a new range updates everything — hero gauge, narrative, cards, tables, and chart. The Custom button opens a date picker for arbitrary date ranges.

Use the 7d view to spot recurring patterns. If CPU spikes at the same time every day, it’s probably a scheduled process running during a busy period. The Process Breakdown table will confirm which process is responsible.

7Top API Endpoints

A ranked table of the most-called Control Center API endpoints, showing which dashboard features generate the most server-side work. Blue accent color matches the API card frame.

Columns include the endpoint path, call count, average response time (ms), and maximum response time. Useful for identifying if a specific endpoint is slow (high avg ms) or just heavily used (high calls but low avg ms).

The ranking is by total processing time (calls × average duration), so endpoints that are both popular and slow rise to the top — these are the best optimization candidates if API performance ever becomes a concern.
