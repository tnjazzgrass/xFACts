About This Page

The DBCC Operations page is the window into scheduled database-integrity checking — the automated DBCC CHECKDB family of commands that verify SQL Server databases aren’t silently corrupting.

The left side is the live view: what’s running right now, and what has run today. The right side is the reference view: the schedule (what’s configured to run, and when) and the full execution history.

Most of the time this page is quiet — integrity checks run on a schedule, usually overnight or on weekends, so during the day you’ll often see “no active executions.” That’s healthy. This guide walks through each section and the schedule editor.

1Engine Indicator

One engine card in the upper right represents the process behind this page: the DBCC executor. It works through the queue of due integrity checks, running each one and recording the result.

A green bar means the last run succeeded; red means it failed. The countdown shows time until the next scheduled run. This card updates on its own, independently of the page’s refresh cycle.

Unlike the collector-style engines on other pages that constantly poll a source, the DBCC engine mostly waits — it wakes on its interval, runs whatever integrity checks are due, and goes quiet again. A long countdown with nothing running is the normal daytime state.

2Live Progress

Shows integrity checks that are executing right now, refreshed on the live interval. When a check is running, this panel shows a progress card for it: the operation and database, the check mode and parallelism, which server it’s running on, elapsed time, an ETA, and a percent-complete bar.

Integrity checks — especially a full CHECKDB on a large database — can run for hours, so the progress bar and ETA are genuinely useful for knowing how much longer a maintenance window will be occupied.

Most of the day this panel reads “No active executions,” because checks are scheduled for off-hours. An empty panel here is the normal, healthy daytime state — not a sign anything is wrong.

3Today’s Executions & Pending

A running list of every integrity check that has executed today — completed and in-progress alike. Each row shows the operation, server, database, duration, completion time, check mode, and final status. Failed and errors-found rows are tinted so they stand out.

In the header, the Pending button shows how many checks are queued but haven’t started yet (the count badge appears when there are any). Clicking it opens the pending queue — the operations waiting their turn, with the database, server, and how long each has been queued.

The pending queue exists because checks are claimed and run one (or a few) at a time to avoid hammering a server with simultaneous full scans. A backlog in the queue during a maintenance window is normal; it drains as each check finishes.

4Refresh Badges

This page is a good example of the three refresh rhythms used across the Control Center, and it shows all three at once:

Live (pulsing dot) — Live Progress and Today’s Executions poll on a short interval, so an in-flight check’s progress and a just-finished result appear quickly.

Static (pin) — Schedule Overview loads once when the page opens and doesn’t auto-refresh. The schedule is configuration; it rarely changes, so there’s no reason to re-poll it.

Event (lightning) — Execution History refreshes when the DBCC engine completes a check, since that’s the only moment new history appears.

Matching each section’s refresh style to how often its data actually changes keeps the page efficient — the fast-moving panels stay current, and the slow-moving ones don’t waste queries re-fetching data that hasn’t changed.

5Schedule Overview

The list of servers that have DBCC schedules configured. Each row shows the server, how many databases are scheduled on it, and whether the server’s checking is Active or Disabled. A warning marker appears when one or more databases on that server have a replica override (more on that below).

This is a summary — click any server row to open its full schedule detail (see the next section). It loads once when the page opens (the pin badge), since the schedule rarely changes.

6Schedule Modal & Editor

Clicking a server in the Schedule Overview opens its schedule modal — shown on the mockup. It lists every database on that server with its check mode and, for each of the four operations (CHECKDB, CHECKALLOC, CHECKCATALOG, CHECKCONSTRAINTS), the day and time it’s scheduled to run, or a dash when that operation is off. A warning icon flags any database whose checks are pinned to a specific replica.

For most people this modal is read-only — a clear picture of exactly what integrity checking is configured. For admins, each row has an edit icon (the calendar) that opens the schedule editor for that one database.

The editor is where scheduling actually happens. It has:

Check mode — None, Physical Only, or Full. This controls how thorough (and how expensive) the CHECKDB is. Setting it to None disables CHECKDB for that database — the editor warns you when that’s about to happen.

Per-operation toggles — a switch for each of the four operations, and when one is on, a day-of-week and time-of-day selector for when it should run.

Target replica (Availability Group databases only) — Default, Primary, or Secondary. This lets an AG database’s checks be pinned to run on a specific replica — typically the secondary, so the expensive scan doesn’t load the primary. That’s the replica override the warning icons refer to.

Changes are saved with Save Changes; each modified value is written back and the schedule refreshes.

This scheduling editor is specific to DBCC — it’s a different design from the scheduling modals on the Index Maintenance and DM Operations pages, because integrity checking has its own per-operation, per-database, per-replica needs.

Editing is admin-only, and enforced on the server side — the edit icon only appears for admins, and the save endpoints reject non-admin requests regardless. Changing a schedule changes when real integrity checks run against production databases, so treat it with the same care as any maintenance-window change.

7Execution History

The full history of every integrity check, organized as a collapsible year → month → day tree. It refreshes when the DBCC engine completes a check (the lightning badge), since that’s when new history appears.

Each level rolls up its counts: executions, succeeded, and failed. Click a year to expand its months, a month to expand its days. A day row adds a few more columns — the number of scheduled runs, total duration, and a warnings count for checks that finished but found something worth noting.

“Succeeded” here includes checks that completed and found issues (the errors-found outcome) as well as clean passes — the check itself ran successfully either way. The separate warnings column is what tells you a completed check actually found something. A failed check, by contrast, is one that couldn’t complete at all.

8DBCC Output Detail

Expanding a day row reveals the individual checks that ran that day — each with its operation, server, database, start and completion times, duration, and status. On an Availability Group database, if the check ran on a different replica than the listener, that target server is noted alongside.

A check row that has captured output or an error is itself expandable. Clicking it reveals the DBCC Output — the actual message SQL Server returned, like the classic “found 0 errors and repaired 0 errors” summary line — and, for a failure, the error detail. This is the ground truth: the raw result straight from the engine, without needing to open SQL Server Management Studio.

The output line includes the exact command that ran (options, MAXDOP, and the account that executed it) plus the elapsed time. When you need to prove an integrity check happened and passed — for an audit, or after a scare — this is the record to point at.
