About This Page

The Backup Monitoring page tracks the entire backup pipeline — from the initial SQL Server backup through network copy and AWS upload to eventual retention cleanup.

Four engine indicators drive the data, one for each stage of the pipeline. Active operations show real-time progress with live polling, while pipeline status, queues, retention, and storage update when their respective engine processes complete.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicators

Four engine cards in the upper right represent the four stages of the backup pipeline. Each runs independently on its own schedule, and each has its own colored bar and countdown.

Backup — Collect-BackupStatus.ps1. Queries SQL Server’s backup history tables and records what backups completed since the last check. This is the pipeline’s starting point — it discovers what files need to be moved downstream.

Network — Process-BackupNetworkCopy.ps1. Copies completed backup files from local drives to the network share. Runs on its own cycle and processes whatever the Backup collector has queued.

AWS — Process-BackupAWSUpload.ps1. Uploads backup files to AWS S3/Glacier for off-site retention. Picks up files that have been copied to the network share and uploads them in order.

Retention — Process-BackupRetention.ps1. Cleans up expired backup files from local and network storage based on configurable retention policies. Typically runs once daily.

Green bar means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in. These update via WebSocket independently of the page’s refresh cycle.

2Pipeline Status

Four cards showing the current state of each pipeline stage. Each card displays the process name, a status badge, the time of the last run, and file/size metrics from the most recent execution.

Status badges indicate the current state: Complete (green) means the last run finished successfully. Running (blue) means the process is actively executing. Failed (red) means the last run encountered an error. Idle (gray) means the process hasn’t run recently or is waiting for its scheduled window.

This section refreshes when any engine process completes (lightning bolt badge). The cards give you a quick left-to-right reading of the pipeline — if Backup and Network are green but AWS shows Running, the pipeline is mid-stream and working normally.

Read the pipeline cards left to right as a flow: Backup creates files, Network copies them, AWS uploads them, Retention cleans up. If a downstream stage shows zero files processed, check whether the upstream stage completed successfully — the pipeline is sequential.

Click any pipeline card to open a detail modal showing the last run’s summary (status, timing, file count, size, duration) and a per-file results table. Failed files will display the error message in red beneath the row.

3Active Operations

Real-time view of what’s happening right now across all three active operation types. This section uses live polling (pulsing dot badge) and queries the servers directly for in-progress operations.

Backups In Progress — SQL Server backup operations currently executing. Shows server, database, backup type, a progress bar with percentage, elapsed time, and estimated time remaining. This data comes from sys.dm_exec_requests on each monitored server.

Network Copies In Progress — Files currently being copied from local backup drives to the network share. Shows server, database, filename, file size, copy progress, elapsed time, and ETA. Progress is estimated from bytes transferred.

AWS Uploads In Progress — Files currently being uploaded to AWS S3. Same detail as network copies — server, database, filename, size, progress, elapsed, and ETA.

When no operations are active for a type, it shows a simple “No active” message. During particularly active backup windows, all three sections may be active simultaneously as different stages process different servers’ files.

4Queue Status

Two cards showing how many files are waiting to be processed by the downstream pipeline stages.

Network — Files that have been backed up locally but haven’t been copied to the network share yet. The count shows pending files and the total size. Zero means the network copy process has caught up.

AWS — Files that have been copied to the network share but haven’t been uploaded to AWS yet. A non-zero count is normal during active processing — files accumulate while the uploader works through them sequentially.

Queue counts are expected to be non-zero during and immediately after the backup window. A growing queue that doesn’t decrease over time indicates a downstream process may be stuck or failing — check the Pipeline Status cards and Engine Indicators for that stage.

Click either queue card (when the count is greater than zero) to open a detail modal showing every pending file with backup type, server, database, filename, backup date, and size.

5Retention Status

Two cards showing how many expired backup files are pending cleanup, plus the scheduled retention window.

Local — Expired backup files still on the SQL Server’s local backup drive. These are files past their configured retention period that will be deleted on the next retention run.

Network — Expired backup files on the network share. Same concept — past retention, pending deletion.

The scheduled time shown at the bottom indicates when the retention process is configured to run. Retention typically runs once daily during a low-activity window to avoid competing with backup and copy operations.

Non-zero retention counts are normal between retention runs — files expire throughout the day but aren’t cleaned up until the scheduled window. If counts grow significantly between runs, check whether the retention process is actually executing by looking at its Engine Indicator and Pipeline Status card.

Click either retention card (when candidates exist) to open a slideout panel showing every file eligible for cleanup. Files are grouped by server with expandable accordions down to database and individual file level. Each database shows its retention cutoff date and chain count so you can verify the retention logic before a run executes. Backup type badges (FULL, DIFF, LOG) provide quick visual scanning.

6Storage Status

Visual gauges showing disk utilization on every drive involved in the backup pipeline. Split into two groups: local backup drives on the SQL Servers and the network backup share.

Local Backup Drives — One bar per server showing the backup drive’s used space. Color-coded against per-drive thresholds: green when healthy, yellow when approaching capacity, red when critically low. The label shows the server name and drive letter, with free space and percentage on the right.

Network Storage — The network share used for backup copies. Same gauge format with the UNC path as the label.

The segmented gauge bars use 80 segments for high visual resolution — each segment represents slightly more than 1% of the drive. This makes it easy to spot subtle changes in utilization over time.

Storage filling up is the most common cause of backup failures. If a local drive gauge is yellow or red, check whether retention is running and whether backup sizes have grown unexpectedly. A full drive will cause the next backup to fail, which cascades through the entire pipeline.
