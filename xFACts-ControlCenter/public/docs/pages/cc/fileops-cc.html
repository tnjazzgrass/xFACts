About This Page

The File Monitoring page tracks SFTP file arrivals across multiple servers. Each configured monitor watches for a specific file pattern in a specific directory, with a defined time window and escalation threshold.

When a file arrives, it’s detected. When it doesn’t arrive by the escalation deadline, it escalates. The page shows today’s queue of all monitor activity, configuration management, and detection history.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicator

The single engine card in the upper right represents Scan-SFTPFiles.ps1 — the orchestrator process that scans all configured SFTP servers for file arrivals. It runs on a fixed interval, checking each monitored directory for files matching the configured patterns.

The colored bar and countdown show the health and timing of the scanner. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in.

Every section on this page except the Configuration cards refreshes when this process completes (lightning bolt badges). The scanner is the single data pipeline behind the Daily Queue and Detection History.

2Status

Three summary cards showing the current state of today’s file monitoring activity.

Escalated — Monitors where the expected file has not arrived by the escalation deadline. The card border turns red when non-zero. Each escalation triggers a Teams notification and/or Jira ticket depending on the monitor’s alert configuration. This is the “someone needs to look at this” count.

Monitoring — Monitors that are actively scanning. The check window has started but the file hasn’t arrived yet and the escalation deadline hasn’t passed. This is the “watching and waiting” count.

Detected — Monitors where the expected file arrived successfully. This is the “all good” count. Includes both on-time and late detections (late detections are files that arrived after the escalation deadline but before the end of the day).

3Daily Queue

A table showing every monitor that has been active today, organized into three groups separated by dividers: escalated (top), monitoring (middle), detected (bottom). Each row shows the status badge, monitor name, relevant time, and detected filename if available.

Escalated rows show the escalation time — when the deadline passed. Monitoring rows show the last scan time or the check start time. Detected rows show the detection time and the actual filename that was found.

Click any row to open the Management Console scrolled to that monitor’s configuration. This is the quickest way to check a monitor’s settings when investigating an escalation.

The Scheduled button in the section header shows how many monitors are configured for today but haven’t started scanning yet (their check window hasn’t opened). Click it to see a modal listing the upcoming monitors with their scheduled start times and countdowns.

The Daily Queue resets each day. Monitors only appear once their check window opens. If you don’t see a monitor you expect, check the Scheduled button — it may not have started yet. If it’s not scheduled at all, check the Configuration section to verify the monitor is active and has the right schedule.

4Configuration

Two clickable cards that open the Management Console. The Monitors card shows the total number of configured file monitors. The Servers card shows the number of SFTP servers being scanned. Both are loaded once on page open (pushpin badge) and don’t refresh on the engine cycle.

Click either card to open the Management Console (see section 6). The Monitors card opens to the monitor list; the Servers card opens to the server list.

5Detection History

A collapsible tree showing the complete history of file detections and escalations. Organized by year and month, with counts of distinct monitors, detections, and escalations at every level.

Click a year to expand its months. Each month row shows monitor count, detection count, and escalation count. Click a month to load its daily breakdown. Click a day to open a slideout with the full timeline of events for that date — every detection and escalation with timestamps, monitor names, and filenames.

Escalation counts in the history tree help identify patterns — if the same monitor escalates every Monday, or if escalations cluster around specific dates, the history makes that visible.

6Management Console

A slide-up panel that covers the lower portion of the page. Contains a flip card with two faces: Monitors on the front, Servers on the back. Click the flip button in the title bar to switch between them.

Monitors — A scrollable list of every configured file monitor. Each row shows the monitor name, file pattern, SFTP server, directory, check start time, escalation deadline, priority, active status, and alert subscriptions. Rows are inline-editable — click a field to modify it. Changes are tracked and saved individually per row. An “Add Monitor” button in the title bar creates a new blank row for configuration.

Servers — A list of configured SFTP servers showing the server name, host, port, and credential reference. This is a read-only view of the servers the scanner connects to.

Each monitor row can also have webhook subscriptions for escalation notifications. The webhook configuration allows routing escalation alerts to specific Teams channels or other endpoints on a per-monitor basis.

Changes to monitor configuration take effect on the next scanner cycle. Disabling a monitor mid-day will stop it from scanning, but any existing escalation for that monitor will remain in the Daily Queue until the next day. Verify monitor settings carefully before saving — an incorrect file pattern or directory path will cause the monitor to never detect its file.
