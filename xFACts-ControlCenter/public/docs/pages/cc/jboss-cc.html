About This Page

The JBoss Monitoring page provides a side-by-side view of all three DM application servers, showing JBoss EAP health metrics collected via the JBoss Management API.

Each server card displays real-time status badges, JVM memory usage, thread counts, database connection pool utilization, transaction metrics, HTTP server statistics, and JMS queue depths — all with delta tracking between collection cycles.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicator

The single engine card in the upper right represents Collect-JBossMetrics.ps1 — the process that queries the JBoss Management API on each application server. It makes 9 composite API calls per cycle across 3 servers, collecting JVM metrics, thread pools, database connection pools, transaction statistics, HTTP listener stats, and JMS queue depths.

The colored bar and countdown show the health and timing of the collector. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in.

All server cards refresh when this process completes (lightning bolt badge). Delta tracking between cycles shows how metrics changed since the previous collection.

2Server Cards & Status Badges

Three side-by-side cards, one per Debt Manager application server. Each card shows the server name, role (Primary or Secondary), and a DC badge if it’s the domain controller. The three-column layout gives immediate visual comparison across the fleet.

At the top of each card, status badges show the health of three checkpoints:

HTTP — Whether the server responds to HTTP requests. Green with response time in milliseconds means healthy. Red means the server is unreachable or returned an error. This is the first indicator that something is wrong — if HTTP fails, everything below it is potentially stale.

Service — Whether the Windows service for JBoss EAP is running. Green means the service is up. Red means it’s stopped.

JBoss — The JBoss server state as reported by the Management API. Green (“running”) is healthy. Yellow (“reload-required”) means a configuration change needs a restart to take effect. Red means the server is in an unexpected state.

An Uptime badge shows how long since the JBoss instance was last restarted, formatted in days and hours.

The question mark icon next to the “Application Servers” section title opens an info modal with a detailed overview of what this page monitors and how the data is collected. Each mini-card within the server cards also has its own question mark for metric-specific explanations.

3JVM Heap

A gauge bar showing how much of the JBoss JVM’s allocated heap memory is in use. Displays used/max in megabytes with a percentage.

Color thresholds indicate pressure: green when usage is normal, yellow when approaching capacity (75%+), red when critically high (90%+). The bar gives an instant visual read across all three servers — a server with a significantly higher heap percentage than the others may be under heavier load or experiencing a memory leak.

Below the bar, secondary detail shows non-heap memory usage and OS working set when available.

Delta tracking shows the change since the last collection cycle. A steadily climbing heap that never drops back suggests garbage collection isn’t keeping up with allocation — a potential precursor to OutOfMemoryError.

4Threads

A gauge bar showing the current JVM thread count relative to the peak. The bar fills based on current/peak ratio — a bar near 100% means the thread count is at its historical maximum.

The delta from the previous cycle indicates whether thread count is stable, growing, or decreasing. A steadily growing thread count that never drops back could indicate thread leaks — threads being created but not properly cleaned up.

5DB Pool

Database connection pool utilization showing active connections out of the maximum pool size, with a gauge bar color-coded by usage percentage. Green when under 50%, yellow at 50–80%, red above 80%.

Detail row shows idle connections, peak usage, and average connection acquisition time in milliseconds. A pool near capacity means new database requests may have to wait for a connection to become available. The delta shows whether pool usage is increasing or decreasing.

If any threads are waiting for connections, an amber alert row appears below the gauge bar with the count.

6Transactions

JBoss transaction subsystem counters displayed as individual rows with per-cycle delta tracking.

In-flight — Transactions currently executing. A real-time count, not cumulative. Non-zero is normal during active processing.

Committed — Cumulative count of successful transactions since JBoss startup. The delta (green) shows throughput — how many transactions completed in the last collection cycle.

Timed out — Transactions that exceeded the timeout threshold. Zero is the goal. A nonzero delta (red) often correlates with database performance issues or lock contention.

Rollbacks — Transactions explicitly rolled back. Small numbers are normal. A spike in rollback deltas (yellow) may indicate application errors or resource exhaustion.

Heuristics — Transactions resolved by heuristic decisions (rare in normal operations). Shown with informational styling when nonzero.

7HTTP Server

Undertow HTTP listener statistics showing cumulative counters with per-cycle deltas.

Requests — Total HTTP requests served since startup. The delta shows how many requests were handled in the last collection cycle — essentially the request throughput.

Errors — Requests that returned error responses. Zero is the goal. A delta spike (red) suggests the application is returning error responses to users.

Data sent — Total bytes transferred to clients. The delta shows recent data throughput.

Processing — Cumulative server-side processing time in milliseconds.

Max request — The longest single request processed (high-water mark, not cumulative). Useful for identifying if any request took unusually long.

If the IO worker queue has pending items, an amber alert row appears indicating requests are queuing.

8JMS Queues

A collapsible accordion showing the JMS message queues configured in JBoss. Click the header to expand or collapse the queue list.

The header summary shows the queue status at a glance: “Idle” (green) when no queues have activity, a count of active queues with the message total when processing, or a “stuck” warning (red) when queues have pending messages with no consumers or no delivery activity.

When expanded, queues are split into Active (queues with recent message activity) and Inactive (idle queues, collapsed by default). Each queue row shows messages added, messages delivered, and messages currently pending, all with per-cycle deltas.

Growing pending counts mean messages are accumulating faster than they’re being consumed. The “stuck” detection specifically identifies queues where messages exist but consumers aren’t processing them — a sign that the consuming application may have stopped or disconnected.

9Server Switch

An admin-only feature for changing which DM application server the SharePoint “Debt Manager” navigation link points to. A blue “Users” badge appears on the server card header of whichever server is currently the active target.

For administrators, clicking the Users badge opens a modal with three server buttons. The currently active server is highlighted with a blue border. Click a different server to stage the switch — a confirmation dialog appears explaining that the change affects all users immediately.

The switch operation updates both a GlobalConfig setting and the SharePoint navigation node via the SharePoint REST API. It may take up to 90 seconds to complete (the SharePoint API can be slow). During the switch, the modal locks to prevent concurrent changes, and a status message shows progress.

After completion, the Users badge moves to the new server card and a success message confirms the change with the performing user’s name.

This immediately redirects all Frost Arnett users who click “Debt Manager” in SharePoint. Coordinate with the team before switching during business hours. The typical use case is redirecting to a secondary server during maintenance on the primary.
