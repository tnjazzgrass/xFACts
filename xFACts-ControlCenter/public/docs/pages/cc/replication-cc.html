About This Page

The Replication Monitoring page tracks the health and performance of SQL Server transactional replication across all published databases.

Unlike most Control Center pages, this one does not use live polling. All sections update when the Replication collector completes a cycle (event-driven via the engine indicator).

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicator

The single engine card in the upper right represents the Replication collector process — Collect-ReplicationHealth.ps1. It runs on a fixed interval and gathers agent status, queue depth, latency, and delivery rate from the distribution database.

The colored bar and countdown show the health and timing of the collector. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in.

Every section on this page refreshes when this collector completes (lightning bolt badges). There is no independent live polling on this page — the engine indicator is the single data pipeline behind everything you see.

2Agent Status

One card per replicated publication showing the current state of all three replication agents: Snapshot, Log Reader, and Distribution.

Snapshot Agent — Creates the initial baseline of table data and schema. Normally idle after initial setup. Should only show “Running” during a manual reinitialization.

Log Reader Agent — Continuously reads the transaction log on the publisher and writes committed transactions to the distribution database. Should always show “Running.” If it stops, new changes stop flowing to subscribers.

Distribution Agent — Picks up transactions from the distribution database and delivers them to the subscriber. Should always show “Running.” “Retry” means it encountered an error and is attempting to recover — this is worth watching but not always critical.

Agent status colors: green for healthy running or expected idle, yellow for retry or starting, red for stopped or failed. A single agent in retry status often resolves on its own within a few minutes. Sustained retry or stopped status needs investigation.

3Delivery Rate

A full-width trend chart showing how many commands per second are being delivered from the distribution database to subscribers. Each replicated publication gets its own line.

Healthy replication shows a steady throughput line. Drops to zero could indicate a stopped agent, network issue, or subscriber unavailability. Spikes typically correlate with large transactions or batch operations on the publisher.

Compare the delivery rate chart with the queue depth chart below. If queue depth is climbing while delivery rate is flat or zero, commands are accumulating but not being delivered — that’s the classic sign of a distribution agent problem.

4Queue Depth

Shows the number of commands waiting in the distribution database to be delivered to subscribers. Each publication gets its own line, color-matched to the delivery rate chart.

Under normal conditions, queue depth stays low and relatively flat. A steadily climbing line means commands are arriving faster than they’re being delivered. A sudden spike could indicate a large transaction on the publisher (bulk update, index rebuild) or a temporary subscriber issue.

Queue depth that climbs and stays elevated is the primary indicator that replication is falling behind. Check the distribution agent status and the event log for errors.

5End-to-End Latency

Measures how long it takes for a transaction committed on the publisher to appear on the subscriber. Shown in seconds. Lower is better.

This is the metric that business stakeholders care about most — it answers “how far behind is the replica?” Under normal load, latency should be in the single-digit seconds. During heavy operations (overnight batch processing, large data loads), latency can spike to minutes and recover once the load subsides.

Sustained high latency with normal delivery rate and flat queue depth might indicate a subscriber performance issue (slow disk, high CPU) rather than a replication problem.

The BIDATA build process creates significant replication load each morning. Elevated latency during the BIDATA window is expected and typically resolves within an hour of build completion. Check the Event Log for correlated BIDATA events.

6Event Log

A scrollable log of replication agent events ordered newest-first. Shows agent messages, errors, and status changes as reported by the distribution database.

Agent Filter — The pill buttons above the log let you filter to a specific publication’s agents. “All” shows everything.

Date Picker — Defaults to today. Select a previous date to review historical events for troubleshooting.

Correlated Button — The chain-link button filters the event log to show only events that coincide with BIDATA build activity. Useful for determining whether replication disruptions are caused by the nightly BI refresh.

Warning-colored rows indicate agent retries or errors. Normal operational messages appear in the default color. The event log is your primary diagnostic tool when agent status or chart trends indicate a problem.

7Time Range Selector

Each chart has a row of time range buttons that control how much history is displayed. The options are 1 hour, 4 hours, 12 hours, 24 hours, and 7 days.

The selected range is highlighted. Clicking a different button reloads the chart data for that window. The active range is independent per chart — you can view delivery rate at 1 hour while looking at latency over 7 days.

Shorter ranges are best for diagnosing active issues. Longer ranges reveal patterns — daily cycles, weekly trends, and whether the current behavior is normal or unusual.

Start with 1h to understand the current state, then widen to 24h or 7d to see if today’s pattern matches recent history. If queue depth is elevated now but was elevated at the same time yesterday, it might be a normal daily load pattern rather than a problem.
