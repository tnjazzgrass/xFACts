About This Page

The DM Operations page is the live monitoring window into the consumer archive and shell purge processes.

Six lifetime tiles up top, today’s activity in the middle, full execution history below — with click-through drill-down all the way from a year down to the per-table detail of a single batch.

This guide walks through every section, control, and the slide-out panels that aren’t visible on the main view.

1Engine Indicators

Two cards in the upper right represent the orchestrated processes that feed this page: Archive (the unified consumer archive) and Shells (the shell purge utility). When the orchestrator is driving them, each card shows a colored bar (green for last-run-succeeded, red for failed) and a countdown to the next scheduled run.

Right now both indicators are inactive. The archive script is currently being run manually as it’s observed in production for early behavior. Once it’s registered in Orchestrator.ProcessRegistry and handed off, the indicators will come alive automatically.

The same shared engine-events module powers these indicators across every monitoring page in the Control Center. Behavior here will look identical to Server Health and the other dashboards once the script is orchestrated.

2Lifetime Totals

Six tiles giving the running totals at a glance. Tiles are not clickable — they’re a quick read; the drill-down lives in the history section.

Consumers Archived — Total consumers fully removed since the unified archive started. The subtitle shows the running batch count, and adds an exception note when any have occurred.

Consumers Remaining — The TC_ARCH-tagged consumer pool still awaiting processing. Calculated from a baseline snapshot taken at the start of the run, with subtractive math applied as batches complete — so the number ticks down throughout the day even between baseline refreshes.

Accounts Archived — The total account count corresponding to those archived consumers, with total OLTP rows deleted shown as the subtitle. Useful as a sense of the actual volume of work done; consumers are the unit, but accounts are what most people picture.

Accounts Remaining — Accounts on TC_ARCH-tagged consumers awaiting processing. Same baseline-plus-subtraction math as the consumer count.

Shells Purged — Total shells removed by the standalone shell purge process. Currently zero because the unified archive doesn’t produce shells — this number will grow as naturally-occurring shells (from merges and historical artifacts) get processed.

Shells Remaining — The remaining shell pool. If exclusions have been catalogued from prior scans, the subtitle shows the exclusion count instead of the baseline timestamp.

The remaining counts use a subtractive technique that makes them feel more live than they technically are. The baseline is captured periodically; as batches complete, the page subtracts what the script just removed from the baseline figure. This avoids querying massive parent tables every refresh while still keeping the displayed value in sync with reality.

3Consumer Archive — Today

A row of counters showing what the archive has done so far today: total batches, consumers, accounts, rows deleted, and total runtime. When the script is active, these tick up as each batch completes.

Two extra counters appear conditionally and only when something needs attention — Exceptions (consumers ejected by runtime re-verification, in yellow) and BIDATA Failed (batches where the financial migration failed, in red). On a clean day, neither shows up. The page stays visually quiet when nothing’s wrong.

The section header carries two admin-only buttons (Schedule and Abort) plus the event refresh badge. Non-admin users see the section but not the controls.

4Consumer Archive — History

The full execution history, grouped year → month → day. Year rows show consumer and account totals; month rows expand to a per-day breakdown; day rows expand again to show the individual batches that ran. State is preserved across refreshes — if you have April 2026 expanded and the page refreshes itself, April stays open.

The columns shift slightly between Archive and Shell Purge: the archive history shows Accounts and an Exceptions column; shell purge history doesn’t (shells have neither concept). Day rows that include a failed batch or a BIDATA failure are visually flagged in their parent month’s row.

This is the easiest way to spot a problem after the fact. Failed batches highlight at the day level, BIDATA failures get a warning glyph in the exception column, and clicking down to the individual batch gives you everything that happened in it.

5Day Drill-Down

Click a day row in the month view and it expands inline to show every batch that ran that day — numbered, time-stamped, with mode (Reduced / Full / Manual / Retry), duration, status, BIDATA status, consumer count, account count, rows deleted, and exception count.

Batches that failed get a red row tint. Batches where BIDATA migration failed but the rest succeeded get a yellow tint — those are the recoverable near-shell cases described on the architecture page. Retry batches get a small R badge with a tooltip pointing back to the original batch.

The batch list is fetched on-demand the first time you expand a day, then cached in the browser. Re-expanding the same day during the session reuses the cached data without re-fetching.

6Batch List & BatchDetail Slide-out

Clicking any batch row in the day drill-down opens an extra-wide slide-out panel showing the full Archive_BatchDetail for that batch — the entire ordered delete sequence with per-table results.

The slide-out has two parts. The header summarizes the batch — start/end times, duration, mode, status, BIDATA status, consumer/account/row counts, exception count, table proc/skip/fail tally, and who executed the batch. If the batch is a retry, that’s called out. If it failed with an error message, that surfaces as well.

Below the header, the per-table table shows every delete operation that ran in this batch with its phase prefix (color-coded: blue for account-level A, purple for BIDATA migration AB, orange for consumer-level C, with a separate tone for the UDEF prefixes AU/CU). Columns: Order, Table, Pass description, Rows affected, Duration, Status, Error.

All columns are sortable — click a header to sort ascending, click again to flip. Useful for finding the slowest tables in a batch or all the tables that skipped (zero qualifying rows is normal — not every consumer touches every table).

The combination of sort-by-duration on this slide-out and the day-level batch list above is the fastest way to investigate “which batch was slow and why?” The duration column on the day list points you to the slow batch; the slide-out points you to the slow table within it.

7Shell Purge Sections

The right column mirrors the archive layout for the shell purge process — Today on top, History below. The structure is identical except shell purge has no Accounts, Exceptions, or BIDATA concept, so those columns simply don’t appear.

Currently the shell purge sections show no activity because the unified archive doesn’t produce shells — the consumer record is removed in the same batch as the accounts. Naturally-occurring shells (from consumer merges, historical artifacts of earlier processes) accumulate over time and the standalone shell purge utility processes them on its own schedule.

Same admin controls as the archive side: Schedule and Abort buttons in the Today header, both gated to admin role.

8Schedule Modal

Clicking Schedule (admin-only) opens a paint-mode 7×24 grid where you set each hour of each day to Full, Reduced, or Blocked. Pick a paint color from the buttons at the top, then click or drag to apply it. Changes save automatically when you release the mouse — there’s no Save button to forget.

Three modes:

  • Full (green) — the script runs at the configured full batch size. Reserved for evenings, weekends, and other quiet windows.
  • Reduced (yellow) — the script runs at the configured reduced batch size. Used during business hours to keep volume up without impacting users.
  • Blocked (red) — the script doesn’t run during this hour. Used around the BIDATA build window and any other times the system needs to be undisturbed.

Each process has its own schedule — Archive and Shell Purge can run on different windows. The grid you’re editing is identified in the panel title.

The script honors the schedule at batch boundaries, not mid-batch. A change that takes a cell from Full to Blocked while a batch is in flight has no effect on that batch — it completes normally. The new mode applies to the next batch.

9Abort Button

Clicking Abort (admin-only) sets the GlobalConfig abort flag for the chosen process. The currently-running script checks this flag at every batch boundary — when it sees the flag set, it finishes the in-flight batch cleanly and exits. No consumer is ever left half-archived.

The button is confirmation-gated to prevent accidental triggers. When set, the button label changes to ABORT SET and stays that way until cleared. To resume normal operation, click the same button again to clear the flag.

Setting Abort doesn’t kill an in-flight batch; it stops the script after the current batch finishes. If you need an immediate stop in a true emergency, terminating the running PowerShell session is the heavier option — but understand the in-flight batch’s state will depend on where it was when killed. Abort is the clean stop; session kill is the panic stop.
