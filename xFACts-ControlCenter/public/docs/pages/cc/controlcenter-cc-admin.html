About This Page

The Administration page is the command center for the xFACts platform. It’s restricted to the Applications Team and provides direct control over the orchestrator engine, process scheduling, version tracking, and runtime configuration.

The top half is a real-time process timeline — a visual canvas showing every orchestrator execution as it happens. The bottom half is a row of management cards, each opening a dedicated tool.

This guide walks through each section and every panel that opens from it.

1Process Timeline

The full-width canvas visualization that dominates the top of the page. Every orchestrator-managed process gets a swim lane, and every execution paints a colored bar in real time.

Toolbar — The bar across the top offers three controls. Filter pills on the left (All, Running, Failed) let you isolate what’s visible. The module color legend in the center tells you which color belongs to which module — ServerOps is blue, JobFlow is purple, BatchOps is amber, and so on. The window selector on the right (10m, 30m, 60m) controls how much history the canvas shows.

Process Sidebar — The left column lists every registered process, grouped by dependency group (Collectors, Processors, Scanners, Queue Processors). Each row shows:

Power button — a small green dot (●) when enabled. Click it to toggle the process’s run_mode between ON (normal), OFF (disabled), and QUEUE (completes current run then stops). This is the quick way to disable a single process without opening the Process Scheduler.

Process badge — the name, color-coded by current status. Green = last run succeeded. Blue with a pulse = currently running. Red with a pulse = last run failed.

Daily counts — success and failure counts since midnight. Green number = successes. Red number = failures. Zero counts are dimmed.

Countdown — time until the next scheduled execution. Teal countdown ticks down in real time. Yellow “overdue” means the next run is past due. Blue italic “queued” means the process is waiting for its dependency group to clear.

Canvas Area — The right side renders execution bars on a time axis. Each bar’s horizontal position and width correspond to start time and duration. A vertical teal “NOW” line scrolls steadily to the right. Bars for currently running processes grow in real time and glow blue. Failed executions are red. Hover over any bar for a tooltip with process name, status, start/end times, and duration. Click a bar to open a log modal with the full output and error text.

The timeline updates via WebSocket — you’ll see bars appear and grow without refreshing. A 60-second safety-net poll fills in anything the WebSocket missed (reconnects, brief disconnections).

2Platform Management

The row of seven cards below the timeline. Each card opens a different management tool as a slide-up panel — except Platform Monitoring, which navigates to its own full page.

Engine Controls (⚙) — Orchestrator drain mode and Windows service management. Two status pips in the corner show engine and service health at a glance. See section 3.

System Metadata (📋) — Version tracking tree with a version bumper for recording deployments. See section 4.

Global Configuration (⚡) — Runtime settings editor for every configurable value in the platform. See section 5.

Process Scheduler (🕓) — Schedule editor for all orchestrator-managed processes. See section 6.

Platform Monitoring (📈) — The only card that doesn’t open a panel. It navigates to the Platform Monitoring page, which has its own separate guide. See section 7.

Documentation (📚) — DDL reference generation, Confluence publishing, and file consolidation pipeline. See section 8.

Alert Failures (⚠) — Unresolved alert delivery failures with resend capability. A red badge on the card shows the count when failures exist. See section 9.

3Engine Controls

A slide-up panel with two sections: the orchestrator engine and the Windows service.

Engine Breaker Switch — A physical-style ON/OFF toggle that controls the orchestrator’s drain mode. When ON (green handle, green status light, “ONLINE”), the orchestrator runs processes normally. Flipping it OFF (red handle, red light, “DRAINING”) tells the orchestrator to stop launching new processes and let currently running ones finish. It doesn’t kill anything — it just drains gracefully.

While draining, the breaker plate pulses with an amber glow if the Windows service is still running — a visual cue that you’ve stopped the brain but not the body.

Service Controls — Below the divider line, the service section shows the current state of the xFACtsOrchestrator Windows service (Running, Stopped, or transitional states like StartPending/StopPending). Three buttons — Stop, Start, Restart — are enabled or disabled based on a state machine: you can’t stop something that’s already stopped, can’t start something that’s running, etc. Transitional states disable all buttons until the operation completes.

A guidance message below the buttons provides contextual next-step instructions based on the current combination of engine and service state.

Stopping the service terminates the orchestrator process entirely. All running process executions are interrupted. Use drain mode first to gracefully clear the pipeline, then stop the service when the timeline shows no active bars.

4System Metadata

A slide-up panel with a collapsible tree showing every module and component in the platform, their current version, and a built-in version bumper.

Root level — The top “xFACts” row shows total component and object counts across the entire platform, plus the date of the most recent version entry.

Module level — Click a module name to expand it. Each module shows its component count and total registered objects.

Component level — Each component card shows the name, current version number, and object count. Click a component to expand the version bumper.

Version Bumper — The expanded component view shows the next auto-incremented version number (patch level auto-advances; minor rolls over at .9; major rolls over at .9.9). Enter a description of what changed and click Insert to record the deployment. The Insert button stays disabled until you type a description — no empty version entries.

Version History — A button below the bumper opens a detail panel showing every previous version of that component: version number, description, and deployment date.

Object Catalog — Another button opens a detail panel listing every registered database object, script, route, CSS, and JS file belonging to that component, grouped by category with type badges (TABLE, PROC, SCRIPT, ROUTE, etc.).

Version bumps are the standard way to record deployments. One bump per session per component, with a consolidated description of everything that changed. The version number auto-increments — you only need the module, component, and description.

5Global Configuration

A slide-up panel with a module-grouped tree of every runtime configuration key in the platform.

Module groups — Click a module name to expand its configuration keys. Each module shows its key count.

Configuration rows — Each row shows the key name and current value. Click any value to enter inline edit mode — a text input replaces the display value with save/cancel buttons. Changes are applied immediately via API with a confirmation dialog.

Boolean toggles — Keys with BIT data type render as toggle switches instead of text values. Click to flip between ON and OFF. The toggle updates immediately after confirmation.

Alert mode toggles — Keys ending in “_alert_mode” get a special three-state toggle: OFF (0), Teams only (1), Jira only (2), or Both (3). This controls where alerts for that feature are routed.

Adding new keys — Each module group has a “+” button to add new configuration keys. Specify the key name, value, and data type (varchar, int, bit, decimal, time, date, datetime).

Active/Inactive filtering — Toggle buttons at the top filter between active keys and soft-deleted inactive keys. Inactive keys can be reactivated.

6Process Scheduler

A slide-up panel with a module-grouped tree of every orchestrator-managed process and its schedule configuration.

Process cards — Each process shows its run status (ON/OFF/QUEUE badge), name, execution mode (F&F or WAIT badge), and dependency group number. Click to expand the full configuration view.

Expanded view — Shows the process description, script filename, and an editable configuration card. Editable fields include dependency group, interval in seconds, scheduled time, and timeout. Click any value to edit inline.

Execution Mode toggle — Switches between Fire & Forget (engine launches the process and moves on) and Wait (engine waits for the process to finish before continuing). Changes take effect on the next execution cycle.

Allow Concurrent toggle — Controls whether the process can have multiple instances running simultaneously. Most processes are single-instance — enabling concurrent execution is rare and intentional.

Adding new processes — Each module group has a “+” button. Select a script from the file browser dropdown (auto-populates process name and suggested description), configure the execution mode, dependency group, interval, and timeout, then submit. New processes are always created in the OFF state — enable manually when ready.

The run_mode toggle in the Process Timeline sidebar (section 1) and the ON/OFF badge here are the same setting viewed two different ways. Changing one updates the other.

7Platform Monitoring

The only card in the Platform Management row that doesn’t open a slide-up panel. Clicking it navigates to the Platform Monitoring page — a dedicated full-page dashboard for measuring the resource impact of xFACts on the server environment.

That page has its own server selectors, time range controls, CPU impact gauge, process breakdown tables, API analytics, and trend charts. It’s an analysis tool rather than a management tool, which is why it lives on its own page rather than in a panel.

See the Platform Monitoring Guide for a full walkthrough of that page’s layout and features.

8Documentation Pipeline

A slide-up panel that runs the platform’s documentation generation and publishing pipeline from the Control Center UI.

Pipeline steps — Three checkboxes control which steps run: Generate DDL Reference (regenerates the JSON data files from Object_Metadata), Publish to Confluence (converts HTML pages and publishes via REST API), and Consolidate Upload Files (collects all platform files into the upload folder for Claude project knowledge sync).

Options — A grid of toggles for Publish to Confluence, Export Markdown, Include SQL Objects, and Include JSON. These fine-tune what the pipeline produces.

Execution — Click “Run Selected” to launch. The pipeline runs server-side with status polling. Each step shows a progress indicator (hourglass while running, checkmark on success, X on failure). The overall status bar shows “Pipeline running…” during execution and a final summary when complete.

Output — After completion, each step’s results are shown with exit codes and expandable output/error text. This is the same information you’d see in the PowerShell console, rendered in the browser.

The panel can’t be closed while the pipeline is running — the close button and backdrop click are disabled to prevent accidentally navigating away from an active run.

9Alert Failures

A slide-up panel showing Teams and Jira alerts that failed to deliver. When alerts exist, a red badge on the Platform Management card shows the count.

Failure cards — Each failed alert displays as a card with a module badge (color-coded by source module), a category badge (CRITICAL, WARNING, or INFO), the alert title, the error message explaining why delivery failed, and metadata showing retry count and timestamp.

Resend — Each card has a Resend button. Clicking it opens a confirmation dialog, then queues a fresh copy of the alert for delivery. The button shows “Pending…” while the resend processes, and the card is removed from the list once it succeeds.

When all failures are resolved, the panel shows a green checkmark with “No unresolved alert failures” and the badge on the Platform Management card disappears.

Alert failures typically indicate a transient issue — a Teams webhook timeout, a Jira API blip. Resending usually works. If the same alert keeps failing, the underlying integration may need attention.
