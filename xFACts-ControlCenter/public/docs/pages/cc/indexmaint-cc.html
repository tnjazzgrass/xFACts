About This Page

The Index Maintenance page manages the complete index lifecycle across all monitored SQL Server databases — from registry synchronization through fragmentation scanning to rebuild/reorganize execution and statistics updates.

Four engine processes drive the pipeline. The page provides real-time visibility into active operations, queue depth per database, process health with manual launch controls, and per-database index health with schedule management.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicators

Four engine cards in the upper right represent the four stages of the index maintenance pipeline. Each runs on its own schedule and feeds the next stage.

Sync — Sync-IndexRegistry.ps1. Synchronizes the index registry with the actual indexes on each monitored server. Discovers new indexes, removes dropped ones, and updates metadata. This keeps the registry current so scans and maintenance target the right objects.

Scan — Scan-IndexFragmentation.ps1. Queries fragmentation levels for all registered indexes on each database. Results populate the Index Queue with indexes that exceed the rebuild or reorganize thresholds.

Execute — Execute-IndexMaintenance.ps1. Processes the queue — rebuilding heavily fragmented indexes and reorganizing moderately fragmented ones. Respects maintenance windows and concurrency limits.

Stats — Update-IndexStatistics.ps1. Updates statistics on indexes that were rebuilt or reorganized, ensuring the query optimizer has current distribution data.

Green bar means the last run succeeded. Red means it failed. These update via WebSocket independently of the page’s refresh cycle.

2Live Activity

An adaptive widget showing what’s happening right now. Refreshes on the page’s live polling interval (pulsing dot badge).

When a process is running: Shows the process name with a spinning gear badge, elapsed time, and completed index count. Multiple processes can run simultaneously — each gets its own widget in a stacked layout.

When idle: Shows the most recent completed process with its status badge (Success, Failed, or Partial), name, how long ago it finished, and its duration. This gives you a quick sense of when maintenance last ran and whether it succeeded.

3Process Status

Four cards representing each stage of the index maintenance pipeline. Each card shows a health badge, the process name, and how long ago it last ran. Click any card to open a slideout with detailed results from the last execution.

Health badges: OK (green) means healthy. Running (blue) means actively executing. Delayed (yellow) means overdue. Failed or Stale (red) means the last run errored or the process hasn’t reported in.

For administrators, each card includes a Run button that manually triggers the process. Clicking the button opens a confirmation dialog before launching. This allows on-demand execution outside of the normal schedule — useful for running a scan after a large schema change or triggering maintenance during an unscheduled window.

Manual process launches execute immediately on the target server. Running a full index maintenance pass during business hours can impact query performance due to the I/O and locking involved. The Scan and Sync processes are lightweight and safe to run anytime. Execute and Stats should be reserved for maintenance windows unless the situation warrants it.

4Index Queue

A summary of indexes waiting for maintenance, broken down by server and database. Each row shows the database, the number of indexes needing rebuild (high fragmentation), reorganize (moderate fragmentation), and the total.

Click a row to open a slideout with the full queue detail for that database — every index that needs attention, its current fragmentation level, size, and the action that will be taken (rebuild or reorganize).

The queue is populated by the Scan process and drained by the Execute process. A queue that stays consistently high may indicate that the maintenance window isn’t long enough to process all indexes, or that the scan is finding new fragmentation faster than Execute can address it.

5Active Execution

Real-time view of index rebuild and reorganize operations currently in progress. Refreshes on the page’s live polling interval (pulsing dot badge).

When the Execute process is running, this section shows each index being worked on with its name, table, database, operation type (rebuild or reorganize), progress percentage, and elapsed time.

When no maintenance is running, the section shows a simple idle message. During the maintenance window, this is your real-time view into which specific indexes are being processed.

6Database Overview

One card per monitored database showing the index health summary: total registered indexes, number currently fragmented above threshold, when the last scan ran, and when the last maintenance executed.

Each database card includes a Schedule button that opens a maintenance schedule panel. The schedule defines which days and times maintenance is allowed to run for that database, including holiday exclusions. This is how you control the maintenance windows — ensuring rebuilds only happen during approved times.

The fragmented count is color-coded: green when low, yellow when moderate, red when high relative to the total index count. A database with zero fragmented indexes after a recent maintenance run is in good shape. A database with a climbing fragmented count may need a longer window or more frequent scans.

Compare the fragmented count with the Index Queue totals. If the Database Overview shows 22 fragmented indexes but the queue only has 8, the difference is indexes that are fragmented but below the maintenance action threshold — they’re being monitored but don’t yet need intervention.

7Maintenance Schedule

A slide-out panel opened by clicking the calendar icon (📅) next to any database in the Database Overview. Shows a visual grid of when index maintenance is allowed to run for that database.

Standard Schedule — A 7-day × 24-hour grid where each cell represents one hour. Green cells mean maintenance is allowed during that hour; dark cells mean it’s blocked. The typical pattern is overnight windows (e.g., 7 PM to 6 AM on weekdays) with wider or fully open windows on weekends.

Holiday Schedule — A single row of 24 hours defining the maintenance window for recognized holidays. This overrides the standard schedule on holiday dates. Holidays are typically given wider windows since no users are working.

Click or drag to toggle cells between allowed and blocked. Changes save automatically. Dragging across multiple cells toggles them all to match the first cell you clicked — if you click an allowed cell, everything you drag over becomes blocked, and vice versa.

Schedule changes take effect immediately on the next maintenance cycle. Blocking hours that are currently in an active maintenance window will not interrupt a running rebuild, but the Execute process will stop picking up new indexes once the window closes.
