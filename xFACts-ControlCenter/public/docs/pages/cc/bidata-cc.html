About This Page

The BIDATA Monitoring page tracks the nightly build process that refreshes the Business Intelligence reporting database. The build runs as a SQL Server Agent job with multiple steps, typically starting in the early morning hours.

This page shows whether today’s build succeeded, how each step performed compared to its recent average, duration trends over time, and a complete build history with drill-down to individual executions.

This guide walks through each section so you know what you’re looking at and what the numbers mean.

1Engine Indicator

The single engine card in the upper right represents Monitor-BIDATABuild.ps1 — the orchestrator process that queries SQL Server Agent job history for the BIDATA build job. It runs on a fixed interval, checking for new step completions and build status changes.

The colored bar and countdown show the health and timing of the monitor. Green means the last run succeeded. Red means it failed. A frozen countdown with no color means the orchestrator hasn’t reported in.

All sections except Duration Trend refresh when this process completes (lightning bolt badges). The Duration Trend refreshes when you change the date range (action badge).

2Live Activity

A row of adaptive status cards showing the current or most recent build for today. The cards change based on the build’s state.

When a build is running: Cards show Status (with a spinning gear and “Running” badge), Started (start time), Elapsed (running duration that updates each cycle), Progress (steps completed out of total expected), and ETA (estimated completion based on historical average duration).

When a build has completed: Cards show Status (“Completed” badge), Date, Started, Completed (end time), and Duration (total elapsed time). All cards have a green border.

When a build has failed: Cards show Status (“Failed” badge), Date, Started, Failed At (failure time), and Failed Step (the specific step that errored). All cards have a red border.

If the build was attempted more than once today, an additional Attempts card shows the retry count.

If it’s past the expected build start time and Live Activity shows “No Build Today,” the SQL Server Agent job may not have triggered. Check the Agent job history directly or verify the job schedule hasn’t been disabled.

3Current Build Execution

A step-by-step list showing every step of today’s build execution. Each row displays the step’s status badge (Completed or Failed), step name, start time, end time, duration, and a 14-day variance comparison.

The 14 Day Var column shows how the step’s duration compared to its 14-day rolling average. Green with a negative percentage means faster than usual. Orange/red with a positive percentage means slower. This immediately highlights steps that are taking longer than expected.

If a build is actively running, a final row appears with a spinning gear badge showing the currently executing step number and its elapsed time. This row updates each time the monitor process runs.

When no build has run today, the section shows a “No Build Today” placeholder with a sleep icon.

Individual step durations can vary significantly day to day based on data volume. The variance column helps distinguish between normal variation and genuine performance degradation. A step consistently showing +20% or more over several days may indicate a growing data issue or resource contention.

4Duration Trend

A bar chart showing total build duration over time. Each bar represents one day’s build. Green bars are successful builds; red bars indicate failures. Click any bar to open a slideout with that day’s full build details.

Time range buttons control how much history is displayed: 30 days (default), 60 days, 90 days, or a custom date range. Selecting Custom opens a date picker modal. The chart refreshes when you change the range (action badge).

On days where the build was attempted multiple times (failure then retry), bars show stacked segments representing each attempt’s wall-clock duration.

Below the chart, summary statistics show the average, minimum, maximum, and number of builds in the selected range. These give quick context for whether today’s build duration is normal.

The trend chart is the fastest way to spot drift. If the bars are gradually getting taller over weeks, the build is taking longer — usually because data volume is growing. A sudden spike on one day could indicate a resource contention issue, a missing index, or an unusually large data load.

5Build History

A collapsible tree showing the complete history of all BIDATA builds. Organized by year and month, with build counts and success/failure tallies at every level.

Click a year to expand its months. Each month row shows the total number of builds, how many succeeded, and how many failed. Click a month to see its daily breakdown. Click a day to open a slideout with the full build details for that date — step-by-step execution, timings, and status for every step.

The build count in the section header shows the total number of builds across all history. Days with multiple build attempts (failure + retry) show the attempt count next to the date.
