Schema Overview

DM Operations has two components under the DmOps schema: Archive (account-level archiving with BIDATA migration) and ShellPurge (orphaned consumer cleanup). Each has its own set of operational tables, but they share the same schedule grid pattern and GlobalConfig structure.

Account Archive Process

The archive script (Execute-DmArchive.ps1) runs as a continuous batch loop. Each iteration selects a batch of consumers whose accounts carry the TA_ARCH tag, loads their data into temp tables, executes a 109-step delete sequence, migrates financial snapshots to the reporting warehouse, and logs everything before checking the schedule and starting the next batch.

The script opens a single persistent connection to crs5_oltp at session start and keeps it open for the entire run. This is critical — the temp tables created during batch loading survive across all operations within the session. A separate connection opens and closes per batch for the BIDATA migration, since that targets a different database on a potentially different server.

Batch selection is deliberately unordered. The query that selects consumers for each batch has no ORDER BY clause. This was a performance decision — adding ORDER BY to the selection query took 49 seconds; without it, 2 milliseconds. The tradeoff is that consumers aren't processed in any predictable sequence, but since they all need to be processed eventually, the order doesn't matter.

The Delete Sequence

This is the heart of the archive process. Debt Manager's schema has deep foreign key chains — deleting an account record requires first deleting from every table that references it, and every table that references those tables, and so on. The delete sequence is a hardcoded, FK-validated order of operations that navigates this dependency tree.

The sequence has three phases:

PhaseScopeTables
Phase 1: UDEFsUser-defined extension tables discovered dynamically at runtime~11 (varies)
Phase 2: Account tablesAll standard tables with FK relationships to the account record~98
Terminal deleteThe account record itself, after all children are clear1

Every DELETE operation uses SNAPSHOT isolation to avoid blocking production queries, chunked deletes (default 5,000 rows per chunk) to manage transaction log volume, and automatic deadlock retry with exponential backoff. If any table operation fails, the entire batch stops — no further tables are processed, because the FK chain means later deletes could produce inconsistent results.

Pre-materialized intermediate ID sets. Several tables in the delete sequence don't have a direct foreign key to the account record — they're two or three joins away. Rather than embedding complex subqueries in every DELETE, the script pre-materializes the intermediate ID sets into temp tables at batch start. This turned a single-consumer delete from 120 seconds down to 0.065 seconds at scale — an 1,846x improvement.

Non-Blocking Design

Both the archive and shell purge processes operate on a live production database that over 200 people use during business hours. The single most important design requirement is that these processes must never block a user from doing their job. Every DELETE could potentially interfere with someone running a query, pulling a report, or working an account — so the scripts use a layered strategy to ensure that never happens.

SNAPSHOT isolation. Every DELETE statement executes under SET TRANSACTION ISOLATION LEVEL SNAPSHOT. Under normal database operation, a DELETE takes locks on the rows it's modifying, and any SELECT trying to read those same rows has to wait. SNAPSHOT isolation changes this equation. When SNAPSHOT is active, readers don't take shared locks at all — they read from a point-in-time version of the data maintained in the version store. The result: our DELETEs and user queries can operate on the same tables simultaneously without either one waiting for the other. Users see a consistent view of the data as of when their query started, completely unaware that rows are being deleted underneath them.

DEADLOCK_PRIORITY LOW. Even with SNAPSHOT isolation, deadlocks can still occur when two operations try to modify the same rows — for example, if a DM process updates a record that our script is also trying to delete. When a deadlock happens, SQL Server has to pick a victim. SET DEADLOCK_PRIORITY LOW tells SQL Server to always sacrifice our session, not the user's. The user's operation completes successfully without interruption. Our script catches the deadlock error (SQL Server error 1205), waits 5 seconds, and retries. It will retry up to 10 times before giving up on that chunk. In practice, deadlocks at SNAPSHOT isolation are rare — the retry exists as a safety net.

Chunked deletes. Rather than issuing a single DELETE that removes tens of thousands of rows in one statement (which would hold exclusive locks for the duration and generate a massive transaction log entry), every DELETE is automatically wrapped with DELETE TOP (5000). The script loops, deleting 5,000 rows at a time, until no more qualifying rows exist. Between each chunk, a 100-millisecond pause gives other operations a window to acquire any locks they need. The chunk size is configurable via GlobalConfig.

Version store consideration. SNAPSHOT isolation relies on the version store in tempdb to maintain row versions for concurrent readers. At production batch sizes (25,000 consumers, ~3.5 million rows per batch), the version store will grow during each batch and release after the transaction completes. Production tempdb is sized to accommodate this, but it's worth monitoring during the initial production runs.

The retry error codes. The script recognizes four retryable SQL Server errors: 1204 (lock resource limit exceeded), 1205 (deadlock victim), 1222 (lock timeout), and 3960 (snapshot update conflict). Any other error is treated as a hard failure that stops the batch. After a retry, the isolation level is explicitly reset to READ COMMITTED to prevent a stuck session state on the persistent connection.

BIDATA Migration

The reporting warehouse (BIDATA) is rebuilt every night from production data. Four tables form the financial reporting core: account demographics, payment details, combined account-payment records, and aggregated payment summaries. Each exists as a pair — a "P" (production/live) table and a "C" (completed/archived) table — fronted by a UNION view that the reports read from.

When accounts are archived, their P-table rows must be migrated to the corresponding C tables before the next nightly rebuild. Otherwise, the rebuild would regenerate P rows from whatever degraded data remains in the production OLTP — producing incorrect financial snapshots.

The migration for each table pair is atomic: INSERT into C, DELETE from P, wrapped in a single transaction with count validation. If the INSERT count doesn't match the expected source count, the entire transaction rolls back. All four table pairs must succeed or the batch is flagged for review.

The timing constraint. BIDATA rebuilds at 1:00 AM. The archive schedule blocks processing in the hours before the rebuild to ensure no accounts are left in a partially-processed state when the rebuild runs. A future enhancement will add PII anonymization to this step — scrubbing personal data in the C tables after migration while preserving the financial figures.

Shell Purge Process

The shell purge script (Execute-DmShellPurge.ps1) follows the same batch loop pattern as the archive script but targets a different problem. A "shell" is a consumer record with no remaining accounts — the consumer exists in the system but has nothing attached to it. Shells accumulate from two sources: account archiving (which removes all accounts but leaves the consumer) and consumer merge operations within Debt Manager.

Eligible shells are identified by their placement in the WFAPURGE workgroup — a Debt Manager nightly job moves qualifying consumers there automatically. The purge script selects from this workgroup, validates candidates against the exclusion log, and executes a 58-step consumer-level delete sequence derived from the vendor's own shell deletion procedure.

Why not just delete from the vendor proc? The vendor's stored procedure processes one consumer at a time with no parallelism, no batch control, no scheduling, and no audit trail. The xFACts implementation processes batches of hundreds or thousands, runs within a schedule, logs every operation, and handles exclusions that the vendor proc doesn't account for.

The Exclusion Log

Not every shell can be safely deleted. Some consumers have data in tables that aren't covered by the delete sequence — either because those tables have no foreign key relationship to the consumer record (making orphaned rows acceptable) or because the data has business significance that warrants preservation.

The exclusion log tracks these consumers with reason codes. It was seeded with an initial scan of production data (~116,000 consumers across 7 exclusion categories) and grows incrementally as the purge process discovers new exclusions during batch validation. Excluded consumers are never retried — once logged, they're filtered out of all future batch selections via a session-cached temp table for performance.

Suspense transactions receive special handling. Consumers with records in the suspense tracking tables can be optionally included or excluded via a GlobalConfig toggle, allowing the team to make a policy decision about disposition without changing the code.

Schedule System

Both processes share the same scheduling model: a 7-day × 24-hour grid where each cell has one of three values. "Full" means the process runs at full batch size. "Reduced" means it runs at a smaller batch size to minimize system impact. "Blocked" means it stops and doesn't restart until the next allowed window.

The scripts check the schedule after every batch. If the mode changes from Full to Reduced, the batch size adjusts immediately for the next batch. If the mode changes to Blocked, the script exits its batch loop and terminates. The next launch (via the Orchestrator or manual trigger) picks up where it left off.

The three-state model was chosen over a simple allow/block because the database serves over 200 users during business hours. Running at full speed (10,000-record batches) during the day would create noticeable contention. Reduced mode (100-record batches) keeps the process moving without impacting users. Full mode is reserved for evenings and weekends when the system is quiet.

Performance & Optimization

The archive process went through extensive optimization during development. The original approach — processing one consumer at a time with individual queries per table — took 120 seconds per consumer. The current implementation processes 10,000 consumers in 649 seconds (0.065 seconds each), a 1,846x improvement achieved through four key changes:

OptimizationImpact
Persistent SQL connection with session-scoped temp tablesEliminated per-query connection overhead; enabled temp table reuse
64 FK supporting indexes on crs5_oltpEliminated full table scans during FK validation on DELETE
Pre-materialized intermediate ID temp tablesReplaced deep subqueries with single-join lookups
ORDER BY removal from batch selectionBatch selection from 49 seconds to 2 milliseconds

The shell purge process followed a similar path. Batch selection went from 92 seconds (NOT EXISTS against millions of rows) to under 1 second using the exclusion log pattern. The terminal consumer DELETE went from 121 seconds to 2.2 seconds after 25 FK supporting indexes were added. An overnight test run processed 2.3 million consumers in 9 hours with zero failures.

How Everything Connects

GlobalConfig

Both processes read their configuration from GlobalConfig entries under module DmOps with category Archive or ShellPurge. Settings include target instance, batch sizes (full and reduced), chunk size for delete operations, alerting toggle, and the abort flag used for emergency stops from the Control Center.

ServerRegistry

Each process has an enable flag on the ServerRegistry row for its target server (dmops_archive_enabled, dmops_shell_purge_enabled). The scripts check this on startup and exit immediately if the flag is off. This provides a server-level kill switch independent of the schedule and abort mechanisms.

Teams Integration

Both processes use the shared Send-TeamsAlert function to queue alerts on batch failures. Alerting is controlled by a per-process GlobalConfig flag (alerting_enabled) so it can be suppressed during testing without removing the alert logic.

BIDATA Dependency

The archive process depends on BIDATA for the P-to-C financial migration. The BIDATA nightly build status is tracked in BIDATA.BuildExecution. The archive schedule is designed to avoid running during or immediately before the build window. Shell purging has no BIDATA dependency.

crs5_oltp (Debt Manager OLTP)

Both scripts target crs5_oltp via the AG-aware connection system in xFACts-Helpers.psm1. Read operations (batch selection, ID materialization) use the secondary replica. Write operations (DELETEs) target the primary. When targeting a standalone test instance, both read and write go to the same server via direct connection.
